<section><div><div><h3>Part 3 of Building Workflow Driven .NET Applications with Elsa 2</h3><h4>Executing workflows from JSON files &amp; Workflow Providers</h4><figure><img data-image-id="1*GDcj0rEH9P102mdrS9RaBQ.png" data-width="1444" data-height="619" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*GDcj0rEH9P102mdrS9RaBQ.png"></figure><p>In <a href="https://sipkeschoorstra.medium.com/building-workflow-driven-net-applications-with-elsa-2-part-2-ab0ffae91b95" rel="noopener" target="_blank">the previous part</a>, we setup a solution with an ASP.NET Core web application and configured it to be an Elsa server.</p><p>We’ve seen that we can design workflows with Elsa Studio that are then somehow available to our app.</p><p>But how and where are these workflows stored?</p><p>In this part, we will learn about workflow providers.</p></div></div></section><section><div><div><h3>Workflow Providers</h3><p>Elsa provides an interface called <code>IWorkflowProvider</code> and its up to the various implementations of this interface to provide <em>workflow blueprints</em> to the <em>workflow registry</em>. Let’s go over these definitions first.</p><h4>Workflow Blueprint</h4><p>The workflow blueprint is an in-memory representation of the workflow to execute. It stores a list of activities and their connections. The workflow engine uses this information to determine what activity to execute next once the current activity finished execution.</p><h4>Workflow Registry</h4><p>The workflow registry is a service that is used to list all available workflows. It is used by various other services such as the trigger indexer to index start activities that represent workflow triggers, and it is used by applications to select workflows to execute.</p><p>The workflow registry queries all workflow providers registered with the service container to build up a complete list of available workflow blueprints.</p><p>Let’s go over the built-in workflow providers next.</p><h3>Built-in Workflow Providers</h3><p>The Elsa Core package provides the following workflow providers out of the box:</p><ul><li>Programmatic Workflow Provider</li><li>Database Workflow Provider</li><li>Blob Storage Workflow Provider</li></ul><h4>Programmatic Workflow Provider</h4><p>This provider enumerates all workflows that are written using the Workflow Builder API. These are regular C# classes that implement the <code>IWorkflow</code> interface.</p><p>For example, take the following workflow:</p><pre>public class HelloWorld : IWorkflow<br>{<br>   public void Build(IWorkflowBuilder builder)<br>   {<br>     builder.WriteLine(&quot;Hello World!&quot;);<br>   }<br>}</pre><p>To make this workflow available, it needs to be registered with Elsa:</p><pre>services.AddElsa(elsa =&gt; elsa.AddWorkflow&lt;HelloWorld&gt;);</pre><p>Now, the <code>ProgrammaticWorkflowProvider</code> will know about this workflow and be able to yield a workflow blueprint from it.</p><h4>Database Workflow Provider</h4><p>This provider enumerates all workflows as stored in the database. These workflows are stored in a JSON format and are ultimately de-serialized into workflow blueprints.</p><p>It is this provider that we saw at play in the previous part. Elsa Studio posts workflow definitions to some Elsa API endpoint, which stores them into the configured database.</p><h4>Blob Storage Workflow Provider</h4><p>This provider is similar to the Database workflow provider, but instead of reading workflows from a database, it reads them from JSON files through the <code>IBlobStorage</code> abstraction provided by <a href="https://github.com/aloneguid/storage" rel="noopener" target="_blank">Storage.NET</a>.</p><blockquote>Storage.NET can be configured with a myriad of storage providers, such as local directories, Azure Blob Storage, FTP servers, and much more.</blockquote><p>By default, Elsa configures an in-memory blob storage provider. But in this post, we will see how to configure it with a local directory so that we can store workflows as JSON files local to our application.</p><blockquote><strong>Why workflows as local JSON files?</strong></blockquote><blockquote>There are a variety of reasons why you might consider storing workflows as part of your application instead of a database. One such reason might be to take advantage of versioning through git. Another reason might be to have complete control over when new logic gets deployed to your production environments. After all, workflows too are ultimately application code. Depending on the nature of your application, you may or may not find it desirable to be able to change application logic on the fly.</blockquote><blockquote>In the end, it’s all about having options.</blockquote><h3>Using Workflows as JSON Files</h3><p>We created a simple Hello World workflow in the previous part and now understand that this workflow is stored in a database of our application.</p><p>Let’s see how we can serve this workflow from a local JSON file next. Here’s what we will do:</p><ul><li>Export the workflow from Elsa Studio as a JSON file.</li><li>Store the exported JSON file in a sub directory of our <code>DocumentManagement.Workflows </code>project.</li><li>Configure the <code>BlobStorageWorkflowProvider</code>to read workflow files from the sub directory.</li></ul><h4>Export the workflow from Elsa Studio</h4><p>If you don’t already have Elsa Studio running, do so now by running the following command:</p><pre>docker run -t -i -e ELSA__SERVER__BASEADDRESS=&#39;<a href="http://localhost:5000%27/" rel="noopener nofollow noopener" target="_blank">http://localhost:5000&#39;</a> -p 14000:80 elsaworkflows/elsa-dashboard:latest</pre><p>This will start the Elsa Studio ASP.NET application on port 14000, configured to consume Elsa Server API endpoints exposed by <code>DocumentManagement.Web</code> running at <code><a href="http://localhost:5000." rel="noopener" target="_blank">http://localhost:5000</a></code><a href="http://localhost:5000." rel="noopener" target="_blank">.</a></p><p>Next, go to the Workflow Definitions screen and either edit the Hello World workflow or create a new one as outlined in the previous part. This will take you to the workflow definition editor screen.</p><p>From this screen, you can export your workflow by clicking the little arrow pointing downwards that is part of the Publish button:</p><figure><img data-image-id="1*d4PgfMu27VlO8nUS9OFUIQ.png" data-width="930" data-height="380" src="https://cdn-images-1.medium.com/max/800/1*d4PgfMu27VlO8nUS9OFUIQ.png"></figure><p>Click the Export menu item. This will export the current workflow as a JSON file and by default will be downloaded to your Downloads folder (on Windows).</p><p>The contents of the exported file looks something like this:</p><figure><script src="https://gist.github.com/sfmskywalker/5fd7aeb74c439cd92ef225aaafd75e06?file=hello-world.json.js"></script></figure><h4>Copy the workflow to DocumentManagement.Workflows</h4><p>Copy the JSON file and paste it into a new folder of the <code>DocumentManagent.Workflows</code> project. Name this folder <code>Workflows</code>.</p><figure><img data-image-id="1*QYEGlNUiBj-5qqFtD8hNBA.png" data-width="428" data-height="369" src="https://cdn-images-1.medium.com/max/800/1*QYEGlNUiBj-5qqFtD8hNBA.png"></figure><p>To make sure this file gets copied to the build output directory, change its <code>CopyToOutputDirectory</code> from <code>None</code> to <code>Copy Always</code>:</p><pre>&lt;ItemGroup&gt;<br>  &lt;None Update=&quot;Workflows\hello-world.json&quot;&gt;<br>    &lt;CopyToOutputDirectory&gt;Always&lt;/CopyToOutputDirectory&gt;<br>  &lt;/None&gt;<br>&lt;/ItemGroup&gt;</pre><p>Configure Blob Storage Workflow Provider</p><p>All that’s left is configuring the blob storage workflow provider.</p><p>Open the <code>Extensions/ServiceCollectionExtensions.cs</code> file and add the following lines to the <code>AddElsa</code> method:</p><pre>// Get directory path to current assembly.<br>var currentAssemblyPath = Path.GetDirectoryName(typeof(ServiceCollectionExtensions).Assembly.Location);<br><br>// Configure Storage for BlobStorageWorkflowProvider with a directory on disk from where to load workflow definition JSON files from the local &quot;Workflows&quot; folder.<br>services.Configure&lt;BlobStorageWorkflowProviderOptions&gt;(options =&gt; options.BlobStorageFactory = () =&gt; StorageFactory.Blobs.DirectoryFiles(Path.Combine(currentAssemblyPath, &quot;Workflows&quot;)));</pre><blockquote>Use your IDE to automatically import missing namespaces or take a look at the completed code snippets.</blockquote><p>The result should look like <a href="https://gist.github.com/sfmskywalker/5fd7aeb74c439cd92ef225aaafd75e06#file-servicecollectionextensions-cs-L40-L44" rel="noopener" target="_blank">this</a>:</p><figure><script src="https://gist.github.com/sfmskywalker/5fd7aeb74c439cd92ef225aaafd75e06?file=ServiceCollectionExtensions.cs.js"></script><figcaption>src/DocumentManagement.Workflows/Extensions/ServiceCollectionExtensions.cs</figcaption></figure><p>All were doing here is configuring <code>BlobStorageWorkflowProviderOptions</code> with a blob storage factory method using Storage.NET.</p><h3>Try it!</h3><p>To make sure this works, delete the local <code>elsa.sqlite.db</code> file that you will find in the root of the <code>DocumentManagement.Web</code> project folder. This ensures that we will not see the workflow as stored in the database, but only the one provided by the JSON file.</p><p>Start the <code>DocumentManagement.Web</code>application and then navigate to the <em>Workflow Definitions</em> screen:</p><figure><img data-image-id="1*rlFyEvLP7uW4xahA5YbTIg.png" data-width="820" data-height="276" src="https://cdn-images-1.medium.com/max/800/1*rlFyEvLP7uW4xahA5YbTIg.png"><figcaption>Nothing to see here, as expected</figcaption></figure><p>All good. Now navigate to the <em>Workflow Registry</em> screen next:</p><figure><img data-image-id="1*6CbNfnfeaLQupxiLENx_OQ.png" data-width="820" data-height="335" src="https://cdn-images-1.medium.com/max/800/1*6CbNfnfeaLQupxiLENx_OQ.png"><figcaption>Hello World workflow provided by hello-world.json</figcaption></figure><p>As you can see, the Hello World workflow was discovered.</p><h3>Summary</h3><p>Our application is now configured to provide workflows stored in JSON files. Note that we can still create and manage workflow definitions that are stored in the database. It is now up to you to decide whether you want a given workflow to be part of your project or stored in the database.</p><h3>Next</h3><p>In the <a href="https://sipkeschoorstra.medium.com/part-4-of-building-workflow-driven-net-applications-with-elsa-2-d6699a6a247e" rel="noopener" target="_blank">next part</a>, we will be working on the Document Upload screen of our application and setup Entity Framework Core for persisting Document entities and Storage.NET for storing uploaded files.</p></div></div></section>
