<section><div><div><h3>Part 2 of Building Workflow Driven .NET Applications with Elsa 2</h3><h4>Scaffolding the solution</h4><figure><img data-image-id="1*GDcj0rEH9P102mdrS9RaBQ.png" data-width="1444" data-height="619" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*GDcj0rEH9P102mdrS9RaBQ.png"></figure><p>In <a href="https://sipkeschoorstra.medium.com/building-workflow-driven-net-applications-with-elsa-2-part-1-44e08a9ba94b" rel="noopener" target="_blank">the previous part</a>, we introduced the sample solution we are building. In this part, we will scaffold parts of the solution up to the point where we have a basic Razor pages web app and an Elsa Server up and running that can execute workflows.</p><h3>File -&gt; New Project</h3><p>Or, in other words:</p><pre>dotnet new webapp -n DocumentManagement.Web -o DocumentManagement\src\DocumentManagement.Web -f net5.0</pre><p>That will create a new ASP.NET Web Application with Razor Pages and Bootstrap 4. Although optional, I went ahead and updated to Bootstrap 5 for no other reason than that I simply could not help myself.</p><blockquote>To upgrade to Bootstrap 5, simply <a href="https://github.com/twbs/bootstrap/releases/download/v5.0.2/bootstrap-5.0.2-dist.zip" rel="noopener" target="_blank">download the distribution files</a> and replace the <em>js</em> and <em>css</em> folders in <em>\wwwroot\lib\bootstrap\dist.</em></blockquote><p>In addition to creating the web project, we will also add three more class library projects:</p><ul><li><strong>Core</strong> (for domain models and services)</li><li><strong>Persistence</strong> (for data access services)</li><li><strong>Workflows</strong> (for workflow-specifics such as hosting actual workflow JSON files and custom activities)</li></ul><p>Execute the following commands to generate these projects:</p><pre>dotnet new classlib -n DocumentManagement.Core -f netstandard2.1 --langVersion latest -o DocumentManagement\src\DocumentManagement.Core</pre><pre>dotnet new classlib -n DocumentManagement.Persistence -f net5.0 --langVersion latest -o DocumentManagement\src\DocumentManagement.Persistence</pre><pre>dotnet new classlib -n DocumentManagement.Workflows -f netstandard2.1 --langVersion latest -o DocumentManagement\src\DocumentManagement.Workflows</pre><p>Create a solution file and add all four projects to it:</p><pre>dotnet new sln -n DocumentManagement -o DocumentManagement</pre><pre>dotnet sln DocumentManagement/DocumentManagement.sln add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj DocumentManagement/src/DocumentManagement.Core/DocumentManagement.Core.csproj DocumentManagement/src/DocumentManagement.Persistence/DocumentManagement.Persistence.csproj DocumentManagement/src/DocumentManagement.Workflows/DocumentManagement.Workflows.csproj</pre><p>Now that we have a solution structure, we will build up each project re-iteratively.</p><h4>DocumentManagement.Web.csproj</h4><p>The web app will serve both the UI (using Razor Pages) and execute workflows using Elsa and <a href="https://www.hangfire.io/" rel="noopener" target="_blank">Hangfire</a> for background processing.</p><p>If you run the project now, you should see a default web app. Our goal now is to turn the project into an Elsa Server that can execute workflows and to which Elsa Studio can connect.</p><blockquote>Elsa Studio is a set of client-side web components that provides a UI and a visual workflow designer to manage workflows.</blockquote><p>Add the following project references and packages:</p><p><strong>Project References</strong></p><ul><li>DocumentManagement.Core</li><li>DocumentManagement.Persistence</li><li>DocumentManagement.Workflows</li></ul><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj reference DocumentManagement/src/DocumentManagement.Core/DocumentManagement.Core.csproj DocumentManagement/src/DocumentManagement.Persistence/DocumentManagement.Persistence.csproj DocumentManagement/src/DocumentManagement.Workflows/DocumentManagement.Workflows.csproj</pre><p><strong>Package References</strong></p><ul><li>Hangfire.AspNetCore (for hosting Hangfire workers in ASP.NET Core)</li><li>Hangfire.SQLite (for persisting Hangfire jobs in the SQLite database)</li><li>Storage.Net (for abstracting away file access to store uploaded files)</li><li>Elsa.Persistence.EntityFramework.Sqlite (for persisting workflow instances in the SQLite database)</li><li>Elsa.Server.Api (for exposing API Endpoints to Elsa Studio)</li></ul><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj package Hangfire.AspNetCore</pre><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj package Hangfire.SQLite</pre><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj package Storage.Net</pre><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj package Elsa.Persistence.EntityFramework.Sqlite</pre><pre>dotnet add DocumentManagement/src/DocumentManagement.Web/DocumentManagement.Web.csproj package Elsa.Server.Api</pre><h4>Startup</h4><p>Update the Startup class by replacing its contents with the following code:</p><figure><script src="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b?file=Startup.cs.js"></script></figure><p>The code with comments should be mostly self-explanatory, but let’s go over some of the more interesting parts.</p><p><a href="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b#file-startup-cs-L44" rel="noopener" policy.AllowAnyHeader().AllowAnyMethod().AllowAnyOrigin().WithExposedHeaders(&quot;Content-Disposition&quot;)));</pre><p>This registers CORS services, which is necessary only if we want to be able to connect a client applicationthat is hosted on a different origin than the one from which Elsa Server is hosted.</p><p>In our case, we want to be able to run Elsa Studio from a Docker container, which therefore will have a different origin.</p><p><a href="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b#file-startup-cs-L63" rel="noopener" target="_blank"><strong>Line activity before, and it won’t actually be used in any of the three demo workflows. But we will use it to create a very simple “Hello World” workflow in this part.</p><p><a href="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b#file-startup-cs-L67" rel="noopener" target="_blank"><strong>Line 67</strong></a>:</p><pre>endpoints.MapControllers();</pre><p>Although our web application itself won’t expose any API endpoints, the <a href="https://www.nuget.org/packages/Elsa.Server.Api/" rel="noopener" target="_blank">Elsa.Server.Api</a> package does provide a number of API controllers that are consumed by Elsa Studio.</p><blockquote>With the upcoming .NET 6, the API endpoints are likely to be refactored to be using <a href="https://dotnetcoretutorials.com/2021/07/16/building-minimal-apis-in-net-6/" rel="noopener" target="_blank">minimal APIs</a>. This enables greater granularity when it comes to the host application controlling what API endpoints are exposed and what are not.</blockquote><p><a href="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b#file-startup-cs-L93" rel="noopener" target="_blank"><strong>Line 93</strong></a>:</p><pre>services.AddWorkflowServices(dbContext =&gt; dbContext.UseSqlite(dbConnectionString));</pre><p>The <code>AddWorkflowServices</code> extension method doesn’t currently exist, but we will implement this when we work on the <code>DocumentManagement.Workflows</code> project. This method will be setting up Elsa services and activities.</p><p><strong>Configuration</strong></p><p>The startup code references <code>Configuration</code> in a number of places, so let’s update <em>appsettings.json</em> next as follows:</p><figure><script src="https://gist.github.com/sfmskywalker/d466148d58281198b25307f8f5a3e42b?file=appsettings.json.js"></script></figure><p>Notice that the <code>Elsa:Server:BaseUrl</code> setting is configured with a specific URL — this URL must match the same URL with which your application is launched.</p><blockquote>The <code>BaseUrl</code> is used when Elsa needs to generate absolute URLs from background processes where there is no current HTTP context available to infer the base URL. Absolute URLs are used in scenarios such as sending emails containing links to trigger a signal (e.g. Approve &amp; Reject links). If these emails are sent from a background task (for example, when triggered in response to a timer event), we don’t have an actual HTTP context. Having a base URL configured solves this problem nicely. It also helps when hosting your application on containers behind a reverse proxy for example.</blockquote><p>Since we’ll be using the <strong>Send Email</strong> activity to send emails, we require a running SMTP server. We will host one using a Docker container using the following command:</p><pre>docker run -p 3000:80 -p 2525:25 rnwood/smtp4dev:linux-amd64-3.1.0-ci0856</pre><p>This will start <a href="https://github.com/rnwood/smtp4dev" rel="noopener" target="_blank">Smtp4Dev</a>, an awesome little SMTP server ideal for developing applications that need to send email messages which we want to intercept and inspect.</p><p>The SMTP host will listen for instructions at port 2525 and provides a <a href="http://localhost:3000/" target="_blank">web-based UI at port 3000</a>. This UI will help us inspecting email messages without actually sending these messages to anyone.</p><p>Next, we will update the <code>DocumentManagement.Workflows</code>
