This release marks a significant leap forward in our journey towards simplifying and enhancing workflow automation.</p><p><em>What is Elsa Workflows?</em></p><p>For those new to Elsa Workflows, here’s a quick overview: Elsa is a robust suite of NuGet packages designed for building flexible workflow servers. These can be seamlessly integrated within your existing applications or can function as standalone servers. With support for communication via HTTP, AMQP, and other protocols, Elsa is engineered to cater to a diverse range of integration requirements.</p><h3>Known Issues &amp; Limitations</h3><p>Elsa is continually evolving, and while it offers powerful capabilities, there are some known limitations and ongoing work:</p><ul><li>Documentation is still a work in progress.</li><li>The designer is not yet fully embeddable in other applications; this feature is planned for a future release.</li><li>C# and Python expressions are not yet fully tested.</li><li>Bulk Dispatch Workflows is a new activity and not yet fully tested.</li><li>Input/Output is not yet implemented in the Workflow Instance Viewer.</li><li>Starting workflows from the designer is currently supported only for workflows that do not require input and do not start with a trigger; this is planned for a future release.</li><li>The designer currently only supports Flowchart activities. Support for Sequence and StateMachine activities is planned for a future release.</li><li>UI input validation is not yet implemented.</li></ul><h3>What’s New</h3><p>Elsa 3.0 is not just an update; it’s a complete rewrite of Elsa 2 with a host of innovative features and improvements. Here’s a glimpse of what this exciting new release brings to the table:</p><ul><li><strong>New Activity Architecture</strong>: Experience enhanced flexibility with a new activity architecture that supports advanced compositions like <em>Sequence</em>, <em>Flowchart</em>, and the potential for <em>State Machine</em> and <em>BPMN 2.0</em> diagrams.</li><li><strong>Parallel Execution</strong>: Boost efficiency with parallel activity execution, including <em>Parallel</em> and <em>Parallel For Each</em> activities. Additionally, with the introduction of the <em>Task</em> and <em>Job</em> kinds, any activity that you write can execute in parallel to others.</li><li><strong>Enhanced Runtime Options</strong>: In addition to the <em>standard runtime</em>, Elsa 3.0 introduces an <em>Actor model</em> implementation using <a href="https://proto.actor/">Proto.Actor</a> for increased workflow throughput.</li><li><strong>Designer Enhancements</strong>: Design and execute composite activities directly from an improved, user-friendly designer.</li><li><strong>Queue-Based Execution</strong>: Transition from stack-based to queue-based activity execution for breadth-first execution of forks.</li><li><strong>Middleware Pipeline</strong>: Enjoy increased extensibility with the new Workflow &amp; Activity middleware pipeline.</li><li><strong>Intuitive Drag &amp; Drop Designer</strong>: Simplify workflow creation with a drag &amp; drop interface.</li><li><strong>Expanded Expression Languages:</strong> Utilise C# &amp; Python expressions for greater versatility in defining workflows.</li><li><strong>Implicit Fork &amp; Join Support</strong>: Streamline complex workflows with built-in fork and join capabilities.</li><li><strong>Elsa Studio</strong>: Manage workflows efficiently with Elsa Studio, a modular dashboard application powered by Blazor.</li><li><strong>New Activity Types</strong>: Explore new Publish, Event, and RunTask activities for enhanced event and task orchestration.</li><li><strong>Webhooks Module</strong>: Integrate with external applications seamlessly using the Webhooks module.</li><li><strong>Workflow Inbox</strong>: Coordinate events reliably with the Workflow Inbox feature.</li><li><strong>Alterations API</strong>: Modify existing workflow instances at runtime.</li><li><strong>File Management</strong>: Utilise SaveFile &amp; OpenFile activities for interacting with various storage providers, powered by the <a href="https://github.com/robinrodricks/FluentStorage">FluentStorage</a> library.</li><li><strong>Built-In Authentication</strong>: Leverage the default Identity module for immediate authentication and authorization, integrating smoothly with ASP.NET Core’s APIs.</li><li><strong>Persistence Providers</strong>: Benefit from a robust persistence layer with Dapper in addition to Entity Framework Core and MongoDB.</li><li><strong>Improved Workflow Context</strong>: Manage multiple contexts per workflow for greater control.</li><li><strong>.NET Support</strong>: Fully compatible with .NET 6, .NET 7, and .NET 8.</li></ul><p>Elsa 3.0 embodies our commitment to delivering a powerful, flexible, and efficient workflow automation tool. These enhancements are just the beginning of what promises to be a transformative experience for our users.</p><h3>A Brief History</h3><p><em>The Journey from 1.0 to 2.0</em></p><p>Understanding Elsa’s evolution offers valuable insights into its current capabilities. Let’s take a walk down memory lane:</p><h4>1.0: Laying the Foundation</h4><ul><li><strong>Origin</strong>: Elsa 1.0 originated from the <a href="https://docs.orchardcore.net/en/latest/docs/reference/modules/Workflows/">Orchard Core Workflows</a> module, allowing it to be embedded in all types of .NET applications without having a dependency on the Orchard Core framework..</li><li><strong>Visual Designer</strong>: It featured a basic visual designer powered by the <a href="https://jsplumbtoolkit.com/community">community edition of JsPlumb</a>, offering a user-friendly experience.</li><li><strong>Limitations</strong>: Despite its functionality, the designer lacked pan &amp; zoom capabilities, making the management of larger workflows challenging. Moreover, the engine faced hurdles in scaling horizontally, necessitating significant enhancements for future growth..</li></ul><figure><img alt="Elsa 1 workflow designer" src="https://cdn-images-1.medium.com/max/1024/0*fWBJz-TPvLZrn6BZ" /><figcaption>Elsa 1 designer</figcaption></figure><h4>2.0: Advancing the Framework</h4><ul><li><strong>Scaling Capabilities</strong>: Elsa 2.0 introduced distributed locking for horizontal scalability across multiple nodes.</li><li><strong>Redesigned Designer</strong>: A new designer, built on <a href="https://d3js.org/">D3</a> and <a href="https://github.com/dagrejs/dagre">Dagre</a>, was integrated, featuring pan &amp; zoom but lacking drag &amp; drop functionality.</li><li><strong>Challenges</strong>: While the new design improved user interaction, it faced limitations like difficulty in selecting multiple activities and a drop in performance with complex workflows.</li><li><strong>Architectural Limits</strong>: The architecture predominantly supported flowchart-style workflows. While it allowed programmatic creation through a fluent API, the process was cumbersome for larger workflows. Additionally, the support for composite activities, though present, was not without its complexities.</li></ul><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*Dr96ebmsVARBB8o0GKRjdw.png" /><figcaption>Elsa 2 designer</figcaption></figure><p>Elsa 2.0 marked a significant step in the right direction, but it was clear that further advancements were necessary to fully realise the vision of a versatile, scalable workflow engine.</p><h4>3.0: A New Era in Workflow Automation</h4><p><em>Building on Past Learnings</em></p><p>Elsa 3.0 represents a paradigm shift, a version reimagined from the ground up, taking into account the invaluable lessons from Elsa 2.0. This iteration introduces a suite of groundbreaking features and improvements:</p><ul><li><strong>Composable Activity Model</strong>: Drawing inspiration from the Windows Workflow Foundation (WF), Elsa 3.0 introduces a flexible activity model. This allows for the creation of complex workflows using advanced activities like Sequence and Flowchart.</li><li><strong>Revolutionised Workflow Designer</strong>: The designer is now more intuitive than ever, featuring pan &amp; zoom and drag &amp; drop capabilities, significantly enhancing the user experience.</li><li><strong>Actor Model for Lock-Free Execution</strong>: Elsa 3.0 makes use of the Actor model, facilitating lock-free workflow execution. This means more efficient handling of concurrent processes and smoother workflow operations.</li><li><strong>Implicit Fork &amp; Join Constructs</strong>: Simplifying complex workflows, Elsa 3.0 introduces implicit fork and join constructs, streamlining the process of branching and merging workflow paths.</li><li><strong>Versioned Composite Activities</strong>: It is now possible to compose larger workflows with versioned composite activities that can be designed straight from the designer.</li><li><strong>Expanded Expression Capabilities</strong>: With the addition of C# and Python expressions, Elsa 3.0 broadens the horizons for defining dynamic and complex workflow logic.</li></ul><p>In the following sections, we dive deeper into each of these transformative features.</p><h3>Rethinking the Activity Model</h3><p><em>Inspiration and Innovation</em></p><p>One of our primary goals with Elsa 3.0 was to reimagine the activity model. We drew inspiration from the Windows Workflow Foundation (WF) and its dynamic approach to constructing workflows. WF’s model allows for the on-the-fly creation of workflows by instantiating activities and setting their properties, including child activities.</p><pre>var workflow = new Sequence <br>{<br>    Activities = <br>    {<br>        new WriteLine<br>        {<br>           Text = &quot;Hello Windows Workflow Foundation!&quot; <br>        }<br>    }<br>}</pre><p>Now, let’s see how this translates in Elsa 3.0:</p><pre>var workflow = new Sequence<br>{<br>    Activities =<br>    {<br>        new WriteLine(&quot;Hello Elsa 3.0!&quot;) // Used ctor overload to initialize Text property directly<br>    }<br>};</pre><p>The structure remains strikingly similar, maintaining the straightforward and intuitive nature of workflow definition. However, a small enhancement in Elsa 3.0 is the introduction of constructor overloads for commonly used properties, such as the Text property in the WriteLine activity. This refinement simplifies the process of initialising activity properties, making the workflow creation a bit more efficient and user-friendly.</p><h4>Dependency Injection</h4><p><em>Simplifying Activity Creation</em></p><p>We’ve made a significant shift in how activities are created and managed. Unlike Elsa 2, where activities were instantiated via a service container (Dependency Injection or DI), Elsa 3 adopts a more straightforward approach.</p><p><strong>Constructor Flexibility</strong></p><p>Activities in Elsa 3 are simple .NET types, which you instantiate directly to construct a workflow. This change gives activity developers the flexibility to decide which constructors to expose. Best practices suggest having a default constructor and, optionally, overloads for frequently used properties. This shift streamlines workflow composition.</p><p><strong>Accessing Application Services</strong></p><p>You might wonder how activities in Elsa 3 access application services. The process is elegantly simple and mirrors the method used in Windows Workflow Foundation (WF). In Elsa 3, a custom activity can implement an ExecuteAsync method, which receives an ActivityExecutionContext. Services are then resolved from the Service Provider within this context. Here&#39;s a snippet to illustrate:</p><pre>public class MyActivity : CodeActivity<br>{<br>   protected override void Execute(ActivityExecutionContext context)<br>   {<br>      var myService = context.GetRequiredService&lt;IMyService&gt;();<br>   }<br>}</pre><p><strong>Towards a Fluid API</strong></p><p>Although Elsa 3 does not currently have a fluid API for constructing workflows from activities, the architecture is conducive to adding one in the future, potentially supporting property dependency injection.</p><p><strong>Enhanced Scheduling</strong></p><p>A groundbreaking feature of Elsa 3 is the autonomy given to activities to schedule other activities. This capability transforms an activity into a self-contained state machine, enhancing the workflow’s flexibility. In this model, an activity signals its completion, prompting its parent to schedule the next activity based on its logic. This decoupling from the traditional flowchart logic means that the scheduler’s role is now to simply execute the next activity, regardless of its complexity or type, be it a flowchart, sequence, state machine diagram, or potentially a BPMN 2.0 diagram. While the latter two are not yet implemented, the architecture lays the groundwork for their future inclusion.</p><h3>New Designer</h3><p><em>A Fusion of Functionality and Ease</em></p><p>Elsa 3.0 introduces a revamped designer that synthesises the best features of its predecessors while introducing new, user-centric functionalities:</p><ul><li><strong>From Elsa 1 to Elsa 3</strong>: While Elsa 1 broke ground with its drag &amp; drop capabilities, it lacked multi-activity selection, copy &amp; paste functions, and the much-needed pan &amp; zoom feature. Elsa 3.0 not only retains the intuitive drag &amp; drop but also integrates these essential features for an enhanced design experience.</li><li><strong>Advancing Beyond Elsa 2</strong>: The second iteration brought in pan &amp; zoom and an auto-layout using Dagre, significantly improving workflow visualisation. However, it fell short in providing the flexibility to copy &amp; paste activities and drag &amp; drop them into specific locations, a crucial need for managing complex workflows.</li><li><strong>The Best of Both Worlds</strong>: Elsa 3.0 marries the strengths of both previous versions. It boasts a comprehensive set of features including drag &amp; drop, pan &amp; zoom, auto-layout, along with the newly added abilities to copy &amp; paste, and undo &amp; redo actions. This combination ensures a seamless and efficient workflow design process, even for the most intricate workflows.</li><li><strong>Drilling Deeper into Activities</strong>: A unique addition in Elsa 3.0 is the ability to ‘drill into’ activities. This feature allows users to delve into activities that expose properties, which are activity types themselves, enabling a deeper level of customisation and control. This functionality opens up new possibilities for grouping and organising activities, making it easier to manage and understand complex workflows.</li></ul><p>Elsa 3.0’s designer represents a significant leap forward. It’s designed to cater to the needs of both novice users and experienced workflow designers, making it a versatile tool for a wide range of applications.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*2gbjxEhpv5AF5tg3T8HmaA.png" /><figcaption>Elsa 3 designer</figcaption></figure><h4>Enhanced Designer Capabilities: Composable Activities</h4><p><em>Encapsulating Complexity with Ease</em></p><p>A key innovation in Elsa 3.0 is the enhanced support for composable activities within the designer. This feature fundamentally changes how workflows are constructed, allowing for a more intuitive and efficient design process.</p><p>For example, consider a high-level orchestration of a change request process. Something like this, perhaps:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*lQ7licwCwPrCrHyq58IQ5Q.png" /><figcaption>A 3-step change request process</figcaption></figure><p>Each step is a <em>Container</em> activity that itself performs no logic, other than <em>scheduling the activity</em> assigned to its Body property.</p><p>The Body property, being of type IActivity, can support any kind of activity that implements this interface. When using the designer, it will automatically create a new <em>Flowchart</em> activity and assign it to the Body of the Container activity. The flowchart in turn can contain activities that implement a given step in the process. For example, the “Review Change Request” flowchart of the second step might look like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*uVNDfqGivGObnKsfnnPJbg.png" /><figcaption>The contents of the “Review Change Request” container activity</figcaption></figure><h4>Design workflows as activities</h4><p><em>Intuitive Reusability</em></p><p>Elsa 3.0 takes the concept of reusable workflows to the next level by allowing workflows to be designed as activities directly within the designer. This innovation not only simplifies the creation process but also automatically versions composite activities for better management and evolution over time.</p><p>For example, the following represents a workflow that returns the sum of its inputs:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*-i1QjsbW8XaXXoVIB-N73g.png" /><figcaption>A workflow definition called “Sum” that is useable as an activity</figcaption></figure><p>The above screenshot demonstrates an implementation of a workflow called <em>Sum (1) </em>that is <em>usable as an activity (2) </em>under the category <em>Math (3).</em></p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*3O4C-bHC4VzMmhOCr2Mfsw.png" /><figcaption>The Sum workflow accepts one input called Values (an array) and one output called Result</figcaption></figure><p>When the workflow executes, it will execute the RunJavaScript activity, which returns the sum of the values provided as input, which gets <em>bound to the Result output</em> of the workflow.</p><p>With this setup, we can now <em>reuse this workflow as an activity in other workflows</em>. For example, let’s take a look at the following workflow called <em>Calculator</em>:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*TwRag76Z7Wk_1J_prmPsbA.png" /><figcaption>Invoking the Sum workflow as an activity</figcaption></figure><p>This workflow uses the Sum workflow as an activity and captures its Result output into a variable, which can then be used by the WriteLine activity to print out the sum. When running this workflow, the result looks like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*j03qVNao-aM69id_qIPq6g.png" /><figcaption>The workflow instance viewer of the Calculator workflow</figcaption></figure><p>As you can see, the <em>Sum activity (1)</em> received its input values via its <em>Values input property (2)</em> and produced an output of 88 via its <em>Result output property (3)</em>. Notice also that this output was captured in a <em>workflow variable called Sum (4)</em>, which is then used by the WriteLine activity to print out the result to the console.</p><p>Workflows executing as activities run synchronously and don’t produce speearate workflow instances. In fact, what happens under the hood is that the engine simply takes the activities and schedules them for execution.</p><p>Elsa 3.0’s approach to designing workflows as activities offers a flexible and powerful way to construct complex processes with modular components that are easy to manage, version, and integrate into broader systems.</p><h3>Implicit Join &amp; Fork</h3><p><em>Simplified Workflow Branching</em></p><p>Elsa 3.0 introduces a significant simplification to workflow design with its implicit join and fork capabilities:</p><ul><li><strong>Effortless Branching</strong>: Elsa 3.0 allows for the creation of multiple outgoing connections from a single activity, enabling natural and intuitive workflow branching.</li><li><strong>Intuitive Merging with Implicit Join</strong>: Merging these branches is just as straightforward. The new implicit join functionality automatically synchronises converging paths, waiting for all inbound activities to complete before proceeding. This ‘Wait All’ join mode ensures that the subsequent activity only executes once every branch has reached the join point, guaranteeing a synchronised continuation of the workflow.</li><li><strong>Flexible Execution with Explicit Join</strong>: For scenarios requiring a ‘Wait Any’ join mode, Elsa 3.0 provides an explicit Join activity. This mode is designed to resume workflow execution as soon as any one of the inbound activities completes, clearing any bookmarks associated with the remaining paths. This helps with scenarios where you want to wait for one event or the other before continuing.</li></ul><p>Elsa 3.0’s approach to forking and joining represents a leap towards more agile and maintainable workflow designs, providing a more natural way to model concurrent processes and their synchronisation.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*7OzenCfwGvjMh4zE29Ejaw.png" /><figcaption>Implicit Fork &amp; Join</figcaption></figure><p>If you need a <em>Wait Any</em> join mode instead, add an explicit Join activity and configure its Mode to <em>Wait Any</em> (the default for this activity).</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*a5HsblJQ3eBpuD-YQ7OwmA.png" /><figcaption>Explicit Join (Wait Any)</figcaption></figure><p>The <em>Wait Any</em> join mode will continue workflow execution as soon as one of its inbound activities complete, and will automatically clear any bookmarks created by any of its inbound activities. In the above example, if the Event arrives before the Timer triggers, the Timer will be canceled.</p><h3>More expressions: C# &amp; Python</h3><p><em>Broadening the Horizons</em></p><p>Elsa 3.0 introduces support for two powerful and widely-used programming languages:</p><ul><li><strong>C# Expressions</strong>: Leveraging the robustness of <a href="https://github.com/dotnet/roslyn">Roslyn</a>, the .NET compiler platform, Elsa 3.0 allows the use of C# expressions directly within workflows. This opens up a familiar syntax for .NET developers, making it possible to script complex logic without leaving the comfort of the Elsa environment.</li><li><strong>Python Expressions</strong>: With the integration of <a href="http://pythonnet.github.io/">Python.NET</a>, Elsa extends its scripting prowess to include Python expressions. This addition capitalises on Python’s simplicity and extensive module ecosystem, enabling the execution of Python code and the import of modules right within your workflows.</li></ul><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*zgjgkGsTuc9gI8t9aQJC8Q.png" /><figcaption>An example of a C# expression used in the Text input for a WriteLine activity</figcaption></figure><p>Both languages can be enriched with custom types and functions, similar to how it is done in JavaScript and Liquid.</p><h3>Lock free Workflow Execution</h3><p><em>Dual Runtime Implementations for Maximum Flexibility</em></p><p>Elsa 3 ships with two runtime implementations:</p><ul><li><strong>Distributed Locking Runtime</strong>: This default mode, known as the Default Workflow Runtime, relies on a tried-and-tested mechanism that synchronises workflow executions across multiple instances, preventing race conditions and ensuring data integrity.</li><li><strong>Actor Model Runtime</strong>: In contrast, the cutting-edge ProtoActor runtime introduces a lock-free execution model. Because it utilises the third-party library <a href="https://proto.actor/">Proto.Actor</a>, it is provided as an optional module leveraging Elsa’s hooks to harness the full potential of this model.</li></ul><h4>The Power of Virtual Actors</h4><ul><li><strong>Grain-Based Workflows</strong>: With the Actor Model, each workflow instance is treated as a ‘Grain’, a virtual actor with its own message inbox. This structure allows for the isolated, single-threaded processing of messages, ensuring that even if multiple processes attempt to run or resume a workflow, message handling remains orderly and consistent.</li><li><strong>Single-Threaded Assurance</strong>: The actor model guarantees that only one message will be processed at a time for any given workflow instance, eliminating the complexity and overhead of distributed locks and providing a robust, scalable solution for high-concurrency environments.</li></ul><p>Elsa 3.0’s adoption of lock-free execution via the Actor Model opens up new possibilities for performance and scalability, positioning Elsa as a cutting-edge platform for building and managing complex workflows in modern applications.</p><h3>Giving Elsa a Spin</h3><p>If you want to have a quick look, you can follow these steps to try it out using a Docker image.</p><h4>1. Run Elsa Studio + Server Docker Image</h4><p>Run the following command to pull the latest version of the Elsa Docker image and run it on port 13000:</p><pre>docker pull elsaworkflows/elsa-server-and-studio-v3:latest<br>docker run -t -i -e ASPNETCORE_ENVIRONMENT=&#39;Development&#39; -e HTTP_PORTS=8080 -p 13000:8080 elsaworkflows/elsa-server-and-studio-v3:latest</pre><blockquote>Make sure you have <a href="https://www.docker.com/products/docker-desktop/">Docker for Desktop</a> installed before attempting to run Docker images.</blockquote><h4>2. Login</h4><p>Open a web browser window and navigate to <a href="http://localhost:13000/">http://localhost:13000/</a>, then login with the default admin credentials, which are:</p><pre>Username: admin<br>Password: password</pre><h4>3. Import the Sample Workflow</h4><p>First, <a href="https://raw.githubusercontent.com/elsa-workflows/elsa-documentation/main/v3/public/guides/script-types/payload-transformer.json">download this sample workflow</a>.</p><p>Then navigate to <em>Workflows -&gt; Definitions</em> and click the little arrow on the <em>Create Workflow</em> button and select <em>Import</em>.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*GxzXL4bs4IuxDXUFUppw6g.png" /><figcaption>Click the arrow</figcaption></figure><p>Select the workflow file you just downloaded. A new workflow appears in the list.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*5AFR0b92RNuQTGKyKh7NSQ.png" /><figcaption>The imported workflow appears in the list</figcaption></figure><h4>4. Publish the Workflow</h4><p>Before we can run the workflow, make sure it is <em>published</em>. To do so, click on the triple dots at the far right of the workflow and select <em>Publish.</em></p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*rHvSiCIToPwMZZZqDjnZAA.png" /><figcaption>Publish the imported workflow</figcaption></figure><h4>5. Run the Workflow</h4><p>Now that the workflow is published, we can invoke it.
