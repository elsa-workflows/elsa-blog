<h3>Part 7 of Building Workflow Driven .NET Applications with Elsa 2</h3><h4>Putting it all together</h4><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*GDcj0rEH9P102mdrS9RaBQ.png" /></figure><p>In the <a href="https://medium.com/codex/part-6-of-building-workflow-driven-net-applications-with-elsa-2-9b3167c612dd">previous part</a>, we implemented a small number of custom activities that we will put to good use here.</p><p>In this part, we will be putting everything together by creating three workflows, one for each type of document:</p><ul><li>Change Request Workflow</li><li>Leave Request Workflow</li><li>Identity Verification Workflow</li></ul><p>The main purpose of setting up these workflows is to demonstrate how to deal with things like:</p><ul><li>Passing information into the workflow</li><li>Passing information between activities</li></ul><p>And as a bonus we will also see how to:</p><ul><li>Add custom JavaScript functions</li><li>Add custom JavaScript Type definitions for intellisense</li><li>Handle the HTTP response when a user clicks an expired signal URL</li></ul><p>Let’s get started!</p><h3>Change Request Workflow</h3><p>The Change Request workflow will be a simple, sequential workflow that looks like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/520/1*OK8f2P2atHNvCqC6QBJd8g.png" /><figcaption>Change Request Workflow</figcaption></figure><p>This workflow will:</p><ol><li>Load the document and file from the database.</li><li>Compress the file.</li><li>Send it as an attachment via email.</li><li>Mark the document as Archived.</li></ol><h4>Workflow Settings</h4><p>The first thing we do is configure the <em>Tag </em>property of the workflow and associate it with the <em>Change Request</em> document type ID:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/919/1*gCqjHz2H3cpEv-peEHtKTQ.png" /><figcaption>Associate the Change Request workflow with the “ChangeRequest” document type ID.</figcaption></figure><p>Also be sure to give the workflow a proper name:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/916/1*QBF07vEB9KKkhd0s-p9BiA.png" /><figcaption>It’ a good idea to at least provide a name for the workflow.</figcaption></figure><h4>Get Document</h4><p>The first activity to add is the Get Document activity. When doing so, you get to provide the ID of the document to get.</p><p>As you may recall, we have a handler in our application called StartDocumentWorkflows which will execute any &amp; all workflows that are tagged with the same document type ID as the created document .</p><p>When the matching workflows are invoked, we pass in the <em>document ID</em> as the <em>correlation ID</em>.</p><p>This means that we can simply provide this correlation ID as the value for the document ID to get using a simple JavaScript expression:</p><pre>correlationId</pre><figure><img alt="" src="https://cdn-images-1.medium.com/max/913/1*_yk7zwg1agzHUybxj1SwTw.png" /><figcaption>The `correlationId` JS variable contains the document ID value.</figcaption></figure><p>Later in the workflow, we will need to access the loaded document + file, so make sure to give this activity a name as well:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/905/1*tgtxAyzLeUP-y3NKUGgjtg.png" /><figcaption>Named activities can be referenced from other activities.</figcaption></figure><p>Giving an activity a name allows us to access its output from other activities, as we will see next.</p><h4>Zip File</h4><p>Connect the Zip File activity and configure its <em>Stream</em> and <em>File Name</em> properties.</p><p>The <em>Stream</em> property should get the following JavaScript expression:</p><pre>activities.GetDocument1.Output().FileStream</pre><p>And the File Name property should get the following JavaScript expression:</p><pre>activities.GetDocument1.Output().Document.FileName</pre><p>As you type in the above expressions, notice that there is intellisense support to help you discover the activities that are available as well as their properties:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/904/1*cE8nFKCRE25NDfEe5CGCgA.gif" /><figcaption>Activity output is supported by Intellisense.</figcaption></figure><p>Except for the Output property, which is of type DocumentFile . Intellisense doesn’t seem to pick up on that, showing any as its type instead.</p><p>To make intellisense become aware of custom .NET types, we need to register them explicitly. Let’s do that right away.</p><p>Ceate a new folder called JavaScript in the Scripting folder of the DocumentManagement.Workflows project and create the following file:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/0ac2111addf73b104a98fa6828141b2b/href">https://medium.com/media/0ac2111addf73b104a98fa6828141b2b/href</a></iframe><p>Then register this class with the service container from the ServiceCollectionExtensions class in the same project:</p><pre>// Register custom type definition provider for JS intellisense.<br>services.AddJavaScriptTypeDefinitionProvider&lt;CustomTypeDefinitionProvider&gt;();</pre><p>Restart your application and refresh the designer. When you now start typing, intellisense will recognize the return type of the Output property:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/909/1*VtP0X1IP5kN9LVKjB5iakw.png" /><figcaption>Use custom type definition providers to add intellisense support for custom types</figcaption></figure><p>Before moving onto the next activity, be sure to give the <em>Zip File</em> activity a name of ZipFile1 .</p><h4>Send Email</h4><p>The <em>Zip File</em> activity produces a zipped stream as its output, which we can attach to an email and send it over.</p><p>Connect a new <em>Send Email</em> activity and provide the following settings:</p><ul><li>To: policy@acme.com</li><li>Subject: Please review this change request</li><li>Body: Please review attached change request.</li></ul><p>For the Attachments field, we need to return one or more EmailAttachment objects. In this example, we will return just one attachment by entering the following JavaScript code:</p><pre>const fileName = `${activities.GetDocument1.Output().Document.FileName}.zip`;</pre><pre>const zipStream = activities.ZipFile1.Output();</pre><pre>const contentType = contentTypeFromFileName(fileName);</pre><pre>const attachment = new EmailAttachment(zipStream, fileName, contentType);</pre><pre>return attachment;</pre><p>In the above code snippet, we access output from both the <em>Get Document</em> and the <em>Zip File</em> activity.</p><p>We then instantiate a new EmailAttachment object that receives the zipped stream, the file name and the content type.</p><h4>Archive Document</h4><p>We end the workflow by connecting a new Archive Document activity with the following settings:</p><ul><li>Document (JS): activities.GetDocument1.Output().Document</li></ul><p>This activity will mark the document as archived and update it in the database.</p><h4>Testing the Workflow</h4><p>Make sure to publish your changes, then upload a new document with the <em>Change Request</em> selected as the document type.</p><p>After a few seconds, you should receive an email with a zip file containing the uploaded document:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*qWN9dM8tfiS_Lbq0IZ6Asw.png" /><figcaption>The email contains one attachment: a zip file with the uploaded document.</figcaption></figure><p>That’s it for this workflow! The next one will be a <a href="https://elsa-workflows.github.io/elsa-core/docs/next/concepts/concepts-workflows#long-running-workflows">long-running workflow</a> where the user needs to click Approve or Reject before the workflow continues.</p><h3>Leave Request Workflow</h3><p>This workflow will look like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*R3i2zzIJQAV8R817u_Sebg.png" /></figure><p>From Elsa Studio, create a new workflow named LeaveRequestWorkflow .</p><h4>Get Document</h4><p>Like the previous workflow, this one too shall start with the <em>Get Document </em>activity configured as:</p><ul><li>Document ID (JS): correlationId</li><li>Name: GetDocument1</li></ul><h4>Send Email</h4><p>Connect a new Send Email activity with the following settings:</p><ul><li>To: hr@acme.com</li><li>Subject: Please review this leave request</li></ul><p>For the <em>Attachments</em> field, enter the following JavaScript code:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/e8544b5e1d03746cb5b8f9b37bdab9c0/href">https://medium.com/media/e8544b5e1d03746cb5b8f9b37bdab9c0/href</a></iframe><p>This looks similar to the Send Email’s Attachment script used in the previous workflow, except for a couple of differences:</p><ul><li>Here, we will send the file itself, not a zipped version of it.</li><li>We determine the content type based on the file name using a function called contentTypeFromFileName .</li></ul><p>This contentTypeFromFileName function however does not exist within Elsa. So let’s create it ourselves!</p><p>Within the JavaScript folder of the Scripting folder of the DocumentManagement.Workflows project, add the following class:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/a8cd3c5ea071892b38744665b0072c9a/href">https://medium.com/media/a8cd3c5ea071892b38744665b0072c9a/href</a></iframe><p>This is similar to what we did when we configured the Liquid templating engine. Only this time, we handle the EvaluatingJavaScriptExpression event so that we can register a custom function.</p><p>The handler depends on IContentTypeProvider, which is provided by the Microsoft.AspNetCore.StaticFiles packages, so let’s add it as a reference:</p><pre>dotnet add DocumentManagement/src/DocumentManagement.Workflows/DocumentManagement.Workflows.csproj package Microsoft.AspNetCore.StaticFiles</pre><p>Then update ServiceCollectionExtensions to register the service with an actual implementation:</p><pre>services.AddSingleton&lt;IContentTypeProvider, FileExtensionContentTypeProvider&gt;();</pre><p>Now we should be able to use the function, which conveniently returns the content type based on a given file name.</p><p>However, when we try it out in the expression editor, we don’t see the function suggested by intellisense:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/867/1*6jWCcmK2hVmxX2OZ1mM01w.png" /><figcaption>Intellisense has no suggestions.</figcaption></figure><p>Although we can still use the function and type it in ourselves, let’s register the function so that it is easier to use next time.</p><p>To do this, we need to handle the RenderingTypeScriptDefinitions event, which gives us an opportunity to enrich the typescript definiton being generated.</p><p>Since we are providing the function in the `ConfigureJavaScriptEngineWithCustomFunctions` class, let’s have it also handle the RenderingTypeScriptDefinitions event:</p><pre>public Task Handle(RenderingTypeScriptDefinitions notification, CancellationToken cancellationToken)<br>{<br>    var output = notification.Output;<br><br>    output.AppendLine(&quot;declare function contentTypeFromFileName(fileName: string): string&quot;);<br><br>    return Task.CompletedTask;<br>}</pre><p>The complete code should look like this:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/fa553b8f8e2af675c528a51edc5a9f05/href">https://medium.com/media/fa553b8f8e2af675c528a51edc5a9f05/href</a></iframe><p>This time around, we will see our custom function being available from intellisense:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/853/1*2oSh9BBXlQ4tvFEfG9sd2A.png" /><figcaption>Intellisense knows about our custom function.</figcaption></figure><p>Sweet!</p><p>Let’s move on to the <em>Body</em> field next.</p><p>Enter the following Liquid markup:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/f42dca1148259e53bbd0af420f574286/href">https://medium.com/media/f42dca1148259e53bbd0af420f574286/href</a></iframe><p>The <em>Body</em> field provides the recipient with two hyperlinks: an <em>Approve</em> and a <em>Reject</em> link. We generate the URLs for these links using the signal_url Liquid filter, which takes as an input the <em>name of a signal</em> and renders a fully-qualified URL that the user can invoke through their browser.</p><p>Invoking a signal URL will trigger the workflow that is blocked on the <em>Signal Received</em> activity that <em>matches the same signal name</em>.</p><p>We need our workflow to receive either an <em>Approve</em> or <em>Reject</em> signal from the user whom was sent an email with the leave request. A great activity to await user input is the Signal Received activity. However, we need two of them: one to listen for the Approve signal and another for the Reject signal.</p><p>If we were to put them one after another, then only one signal would be handled. Which would then cause the workflow to move on to the next Signal Received activity, putting the workflow in a suspended state again until the other signal is received.</p><p>In other words, that won’t work. How do we solve for that?</p><h4>Fork</h4><p>Meet the Fork activity.</p><p>The Fork activity can split workflow execution into zero or more branches. In our case, we want to split into an <em>Approved</em> and a <em>Rejected</em> branch.</p><p>Go ahead and connect a new Fork activity and configure it with the following branch names:</p><ul><li>Branches: [Approved, Rejected]</li></ul><figure><img alt="" src="https://cdn-images-1.medium.com/max/919/1*vWhTCn3cLzTiSkWcRCXCug.png" /><figcaption>A fork in the road.</figcaption></figure><p>When the Fork activity executes, it will schedule all activities that are connected to all of its branches. Which is exactly what we need.</p><h4>Approved Branch</h4><p>Connect a new <em>Signal Received</em> activity to the <em>Approved</em> outcome of the Fork activity and configure it as follows:</p><ul><li>Signal: Approve</li></ul><p>Next, connect a new <em>Write Line</em> activity with the following settings:</p><ul><li>Text: Inform the requester that the leave request was approved.</li><li>Display Name: Approve</li></ul><p>In real world scenarios, you will likely replace this activity with some form of communication to the person requesting their leave, or perhaps by requesting another review from a colleague. But for our purposes, it’s enough to do a simple write to console out.</p><h4>Join</h4><p>We end the Approved branch with a Join activity, which will merge execution back into a single branch.</p><p>The Join activity has a Mode property that can be one of the following values:</p><ul><li>Wait All</li><li>Wait Any</li></ul><p>The Wait All mode will cause the workflow to remain at the Join activity until after ALL inbound branches have executed the Join activity.</p><p>The Wait Any mode on the other hand will continue as soon as ANY inbound branch executes the Join activity.</p><p>For our workflow, we need the user to either select Approve or Reject, and not both. Therefore, configure the Join activity as follows:</p><ul><li>Mode: WaitAny</li></ul><p>Before we continue with the activities that come after the Join, let’s return to the Rejected branch first.</p><h4>Rejected Branch</h4><p>This branch is similar to the Accepted branch in terms of structure. The only differences are the signal to listen for and the text to write to the console.</p><p>Connect a new <em>Signal Received</em> activity to the <em>Approved</em> outcome of the Fork activity and configure it as follows:</p><ul><li>Signal: Reject</li></ul><p>Next, connect a new <em>Write Line</em> activity with the following settings:</p><ul><li>Text: Inform the requester that the leave request was denied.</li><li>Display Name: Reject</li></ul><p>Now connect the Write Line activity to the Join activity we added earlier:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*m98zNmHo-O3gPqGaksrnBA.gif" /><figcaption>Press and hold SHIFT when clicking the source outcome. Then click the destination activity.</figcaption></figure><h4>Get Document</h4><p>Regardless of the user’s input (Accept or Reject), we want to archive the document.</p><p>However, since the workflow went out of memory when it was suspended (due to the Signal Received activities), the loaded document + file stream are no longer in memory. If you recall, we created the Get Document activity’s Output property to use Transient storage.</p><p>For this reason, we will add another Get Document activity to re-load the document into memory.</p><p>Configure it as follows:</p><ul><li>Document ID: correlationId</li></ul><h4>Archive Document</h4><p>Finally, connect a new Archive Document with the following settings:</p><ul><li>Document (JS): input.Document</li></ul><p>Notice that instead of referencing the previous Get Document activity directly, this time we take advantage of a JS variable called input .</p><p>In this context, input is a reference to the output provided by the previously executed activity. Which means that it references the DocumentFile object provided by the Get Document activity.</p><p>The only real difference is that intellisense doesn’t know at design time what the input might be, since it’s possible for multiple inbound activities to provide output.</p><h4>Testing the Workflow</h4><p>To test this workflow, upload another document but this time using the <em>Leave Request</em> option.</p><p>After a few seconds, you should receive an email with the uploaded document attached:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*pyoLRzqwTBGasUUTsswSzw.png" /><figcaption>An email with the uploaded document was received.</figcaption></figure><p>Looking at the Workflow Instances screen, we indeed see a new workflow instance:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*vtEkLWS-qTtIfTlWn8mnzg.png" /><figcaption>A new workflow instance was created.</figcaption></figure><p>When you click the workflow instance, you will be taken to the workflow instance viewer screen. The workflow instance viewer displays a journal representing the activities that have executed. In addition, the workflow designer displays a green border around each activity that has executed.</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*kL5dkZ3cgekeU8jOgpvBaQ.png" /><figcaption>The workflow is currently blocked on two <strong><em>Signal Received</em></strong> activities.</figcaption></figure><p>As you can see, the workflow is currently blocked on two Signal Received activities: one awaiting the Approve signal, the other awaiting the Reject signal.</p><p>To send any of these signals, simply click one of the links received via the email.</p><p>When you click Approve, the console window of the application should display the following text:</p><pre>Inform the requester that the leave request was approved.</pre><figure><img alt="" src="https://cdn-images-1.medium.com/max/849/1*ezCBgy3AN_yMqjncr6PnFw.png" /><figcaption>The console window printed a line of text, showing that the Approved branch continued.</figcaption></figure><p>When you look at the workflow instance again, you will notice that it now has the <em>Finished</em> status.</p><h4>Improvements</h4><p>One important area of improvement is the HTTP response we received when clicking the Approve or Reject link from the email. Right now, we get something like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1023/1*Q5QCLcq7Z8mJ74mIqRlY5g.png" /><figcaption>Clicking Approve or Reject yields a default HTTP response.</figcaption></figure><p>This won’t do when we go to production of course. What we want to display the user instead is some kind of confirmation page.</p><p>Let’s fix this right away.</p><p>First, create a new Razor page in the web project named LeaveRequestApproved.cshtml and add the following content:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/71c43266ad333e80827d4525c1785e34/href">https://medium.com/media/71c43266ad333e80827d4525c1785e34/href</a></iframe><p>Create another Razor page named LeaveRequestDenied.cshtml and add the following content to it:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/1fb31983d55815dded2a1d67f5e8bc00/href">https://medium.com/media/1fb31983d55815dded2a1d67f5e8bc00/href</a></iframe><p>Now that we have these two pages in place, let’s consider how we can display these in response to the user clicking one of the two hyperlinks from the email.</p><p>The hyperlinks are generated using the signal_url Liquid filter, which points to an API controller exposed by Elsa. This controller is responsible for decoding the signal token and then invoking the appropriate workflow.</p><p>What it also does is publish an event. Which is very convenient, because we can handle this event and write a custom HTTP response. In our case, we can simply send an HTTP 302 — Redirect response that redirects the browser to the appropriate Razor page.</p><p>To do this, create a new folder called Handlers in the web project and add the following class:</p><iframe src="" width="0" height="0" frameborder="0" scrolling="no"><a href="https://medium.com/media/0dd882f76a2f75a16191e4892c0671cb/href">https://medium.com/media/0dd882f76a2f75a16191e4892c0671cb/href</a></iframe><p>To get this handler registered with DI, add the following code to Startup:</p><pre>// Register all Mediatr event handlers from this assembly.<br>services.AddNotificationHandlersFrom&lt;Startup&gt;();</pre><p>When we now start the application again, upload a new document and click on <em>Approve</em>, we get the following result:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/903/1*jhyJwFBFqm7vN7FJlS2Q8g.png" /><figcaption>We now see a user-friendly page upon clicking Approve.</figcaption></figure><p>That’s it for this workflow!</p><h3>Identity Verification Workflow</h3><p>The Identity Verification workflow will look like this:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/955/1*fBRQ7cpHDCvHU8cU5P-rmg.png" /><figcaption>The Identity Verification workflow.</figcaption></figure><p>Since this workflow is almost identical to the previous workflow, I will not go over the entire workflow in details. But I will point out two things:</p><ol><li>Make sure to tag this workflow with the IdentityVerification document type ID.</li><li>Configure the <em>Update Blockchain</em> activity as follows:</li></ol><ul><li>File: input.FileStream</li></ul><p>Remember: the previously executed activity (<em>Get Document</em>) will have its output be provided as input into the next activity (Update Blockchain) automatically, which means it’s available via the input JS variable.</p><blockquote>By convention, if an activity output property is named Output , it will be available to the next activity via the input JS variable and the Input liquid variable.</blockquote><h3>Summary</h3><p>This concludes the blog series about building workflow-driven ASP.NET applications with Elsa 2.</p><p>I hope you found it useful and that you see how easy and fun it is to power your applications with workflows.</p><p>Having workflow-driven apps is a powerful feat. As your set of activities, JS and Liquid building blocks grow, adding and updating business rules or even entire processes becomes so much quicker and easier.</p><p>And you always have the choice of whether these workflows are provided from the database, allowing changes to be made in production systems, or to have them provided as part of your application’s code base in the form of JSON files.</p><p>And that’s what Elsa is really all about: to offer a wide range of options to the developer when it comes to enabling workflows in their applications.</p><p>Here, we’ve seen how to empower an ASP.NET application with workflow capabilities. Another approach might have been to have a separate ASP.NET application acting as the workflow server. The Razor app’s backend would then communicate with the workflow server via Elsa API endpoints and possibly with custom API endpoints as well, which internally leverage the Elsa library. Although not discussed in this series, Elsa also supports conductor-style workflows using generic Task and Event activities. The Task activity represents a task to be executed by the client app (e.g. the Razor app), while the Event activity represents an event raised by the client app, causing the workflow on the workflow server to resume execution.</p><p>This setup works very wel in a microservice architecture.</p><p>Anyway. I could go on and on about all the great things you can do with Elsa Workflows, but I will save all that for future posts.</p><p>Stay awesome!</p>
