<section><div><div><h3>Part 6 of Building Workflow Driven .NET Applications with Elsa 2</h3><h4>Custom Activities</h4><figure><img data-image-id="1*GDcj0rEH9P102mdrS9RaBQ.png" data-width="1444" data-height="619" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*GDcj0rEH9P102mdrS9RaBQ.png"></figure><p>In the <a href="https://sipkeschoorstra.medium.com/part-4-of-building-workflow-driven-net-applications-with-elsa-2-d6699a6a247e" rel="noopener" target="_blank">previous part</a>, we did something really cool: we invoked a workflow in response to a domain event. And although that workflow was rather harmless, now that we have the important pieces in play we can do a whole lot more damage :)</p><p>With Elsa’s increasingly rich set of activity libraries coming to a NuGet feed near you, there’s already a lot you can do without writing custom activities. But in many custom workflow driven applications, you are likely to need the ability to write custom activities. That’s what we will be doing in this part.</p><p>In <a href="https://sipkeschoorstra.medium.com/building-workflow-driven-net-applications-with-elsa-2-part-1-44e08a9ba94b" rel="noopener" target="_blank">the introduction</a>, I mentioned the following custom activities:</p><ul><li>Get Document</li><li>Archive Document</li><li>Zip File</li><li>Update Blockchain</li></ul><p>We will go over each activity and implement them one by one.</p><h3>Get Document</h3><p>The purpose of this activity is to get a document and its associated uploaded file into memory so that the workflow can do something with it. For that, it needs only one input: the ID of the document to load. Its output will be a an object of a new record type that carries both the document and the file stream.</p><p>Since this is the first activity being created, I will go over various aspects of it in more detail.</p><p>Start by creating a new <code>Activities</code> folder in the <code>DocumentManagement.Workflows</code> project. Then create a new class called <code>GetDocument</code>.</p><pre>using Elsa.Attributes;<br>using Elsa.Services;<br><br>namespace DocumentManagement.Workflows.Activities<br>{<br>    public class GetDocument : Activity<br>    {       <br>    }<br>}</pre><p>The above code has the absolute minimum requirement to implement an activity by implementing the <code>IActivity</code> interface which is inherited from the <code>Activity</code> base class.</p><p>In order for this activity to be easier to understand by users of the workflow designer, it is recommended to provide a category and a description as metadata by applying the <code>ActivityAttribute</code>on the class:</p><pre>[Action(Category = &quot;Document Management&quot;, Description = &quot;Gets the specified document from the database.&quot;)]<br>public class GetDocument : Activity<br>{<br>}</pre><p>Within the same file, add the following record type above or after the <code>GetDocument</code> class (include the necessary namespaces as well):</p><pre>+ using System.IO;<br>+ using DocumentManagement.Core.Models;</pre><pre>public record DocumentFile(Document Document, Stream FileStream);</pre><h4>IsExternalInit</h4><p>At this point, you might receive a complaint from your IDE that “<em>[t]he predefined type ‘System.Runtime.CompilerServices.IsExternalInit’ must be defined or imported in order to declare init-only setter.</em>”. We have seen this before so we know what to do: just copy the <code>IsExternalInit.cs</code> file from the <code>DocumentManagement.Core</code> project into the root of the <code>DocumentManagement.Workflows</code> project.</p></div></div></section><section><div><div><h4>Input</h4><p>With that out of the way, let’s continue by adding the activity’s input property called <code>DocumentId</code>:</p><pre>+ using Elsa.Expressions;</pre><pre>[ActivityInput(<br>    Label = &quot;Document ID&quot;,<br>    Hint = &quot;The ID of the document to load&quot;,<br>    SupportedSyntaxes = new[] {SyntaxNames.JavaScript, SyntaxNames.Liquid}<br>)]<br>public string DocumentId { get; set; } = default!;</pre><p>The <code>ActivityInputAttribute</code> is necessary for both the engine as well as the designer. The engine uses the attribute as a marker to determine what properties to serialize, while the designer uses the additional metadata such as <code>Label</code> and <code>Hint</code>to display the activity in a user-friendly manner. The <code>SupportedSyntaxes</code> property is leveraged by the activity editor when displaying this property and allows the user to switch from the property’s default input editor to a JavaScript or Liquid editor.</p><h4>Output</h4><p>Next, add the following output property:</p><pre>[ActivityOutput(<br>    Hint = &quot;The document + file.&quot;,<br>    DefaultWorkflowStorageProvider = TransientWorkflowStorageProvider.ProviderName)]<br>public DocumentFile Output { get; set; } = default!;</pre><p>The <code>ActivityOutputAttribute</code> is also used by the workflow engine as well as the designer in a similar way to <code>ActivityInputAttribute</code>. The<code>Hint</code> property isn’t currently leveraged by the designer, but this may change in a future version so it’s good to be prepared.</p><p>The <code>DefaultWorkflowStorageProvider</code> property tells the engine what <em>workflow storage provider</em> to use to persist the value of <code>DocumentFile</code>.</p><p>By default, all activity properties are serialized as part of the workflow instance. But this may not be desirable for all types. For example, our <code>DocumentFile</code> record contains a property of type <code>Stream</code>. Which is usually an indication that something large might be stored here. If we were to serialize it to base64, the workflow instance might become quite large, which would negatively impact performance every time the instance is loaded from the database, which also means it needs to be deserialized.</p><p>By using a different storage provider such as the <code>TransientWorkflowStorageProvider</code>, the object will be stored in-memory only.</p><blockquote>The user can control what storage provider to use on a per-property basis. The <code>DefaultWorkflowStorageProvider</code> merely provides the default setting. If you want to prevent the user from having an option at all, set the <code>DisableWorkflowProviderSelection</code> property to <code>true</code>.</blockquote><h4>Execute</h4><p>When the activity executes, we want it to do the following things:</p><ol><li>Load the document by ID from the database.</li><li>Read the associated file as a stream.</li><li>Return these things as output.</li></ol><p>To load documents, we need to inject <code>IDocumentStore</code>. And to read files, we need to inject <code>IFileStorage</code> . Update the class with the following fields and constructor:</p><pre>+ using DocumentManagement.Core.Services;</pre><pre>private readonly IDocumentStore _documentStore;<br>private readonly IFileStorage _fileStorage;<br><br>public GetDocument(IDocumentStore documentStore, IFileStorage fileStorage)<br>{<br>    _documentStore = documentStore;<br>    _fileStorage = fileStorage;<br>}</pre><p>Next, override the <code>OnExecuteAsync</code> method as follows:</p><pre>+ using System.Threading.Tasks;<br>+ using Elsa.ActivityResults;<br>+ using Elsa.Providers.WorkflowStorage;<br>+ using Elsa.Services.Models;</pre><pre>protected override async ValueTask&lt;IActivityExecutionResult&gt; OnExecuteAsync(ActivityExecutionContext context)<br>{<br>    var document = await _documentStore.GetAsync(DocumentId, context.CancellationToken);<br>    var fileStream = await _fileStorage.ReadAsync(document!.FileName, context.CancellationToken);<br><br>    Output = new DocumentFile(document, fileStream);<br>    return Done();<br>}</pre><p>The complete file should look like this:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=GetDocument.cs.js"></script></figure><h4>Register Activity</h4><p>Before we can actually use the activity on a workflow, we need to register it with Elsa. To do that, update the <code>ServiceCollectionExtensions</code> class and add the following code right after the <code>.AddHttpActivities()</code> invocation:</p><pre>// Add custom activities<br>.AddActivitiesFrom&lt;GetDocument&gt;()</pre><p>This will register not only the <code>GetDocument</code> activity, but also any additional activities we create in this project.</p><h4>Try it out</h4><p>Although we won’t implement any of the document handling workflows until the next part, let’s make sure the activity works by following these steps:</p><ol><li>Start the web application.</li><li>Start Elsa Studio.</li><li>Create a new workflow.</li><li>Add the new <strong>Get Document</strong> activity and set its Document ID setting to the following JavaScript syntax: <code>correlationId</code>.</li><li>Connect a new <strong>HTTP Response</strong> activity with the following Liquid content: <code>The document’s file name is {{ Input.Document.FileName }}</code></li><li>Publish the workflow and take a note of its ID (navigate back to the Workflow Definitions screen, click on the newly created workflow and copy the workflow ID from the browser’s address bar).</li><li>Go to the home page of the web application and upload a document. Take note of the generated document ID.</li><li>Open Postman or your own favorite HTTP client and execute the following request to invoke the workflow using the Elsa API (substitute “{your-workflow-id}” and “{your-document-id}” with your own values) :</li></ol><pre>curl --location --request POST &#39;https://localhost:5001/v1/workflows/<strong>{your-workflow-id}</strong>/execute&#39; \<br>--header &#39;Content-Type: application/json&#39; \<br>--data-raw &#39;{<br>&quot;correlationId&quot;: &quot;<strong>{your-document-id}</strong>&quot;<br>}&#39;</pre><p>The response should look like this:</p><pre>The document&#39;s file name is</pre><p>This is probably not what you expected. Although we clearly &amp; correctly applied a Liquid expression of: <code>The document’s file name is {{ Input.Document.FileName }}</code> , and we know the document is stored in the database with an actual file name, for some reason it does not render the file name.</p><h4>Liquid &amp; Allow-Listing</h4><p>To understand why, we need to know that Elsa uses the <a href="https://github.com/sebastienros/fluid" rel="noopener" target="_blank">Fluid library</a> to evaluate Liquid expressions. And although Fluid is perfectly capable of accessing .NET objects, <a href="https://elsa-workflows.github.io/elsa-core/docs/next/expressions/expressions-liquid#allow-listing-a-specific-type" rel="noopener" target="_blank">it needs to be told what types the user is allowed to access</a>. This makes Liquid a secure templating language.</p><p>Since our liquid expression access both the<code>DocumentFile</code>and <code>Document</code> types (since <code>Document </code>is a property of it), all this means is that we need to allow-list them with Liquid. Let’s do that right away.</p><p>Within the <code>DocumentManagement.Workflows</code> project, create a new folder called <code>Scripting</code> and a subfolder called <code>Liquid</code>. Then create the following class:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=ConfigureLiquidEngine.cs.js"></script></figure><p>Notice that this is a notification handler that handles the <code>EvaluatingLiquidExpression</code> event. This event is published every time the engine is about to evaluate a liquid expression for a given activity property.</p><p>In the event handler, we access Fluid’s <code>TemplateContext</code> and configure its <code>MemberAccessStrategy</code> by registering our <code>DocumentFile</code> type.</p><p>Since we already call <code>.AddNotificationHandlersFrom&lt;StartDocumentWorkflows&gt;()</code> from the <code>ServiceCollectionExtensions</code> class, this event handler will automatically be registered with the service container.</p><p>When we execute the HTTP request again, we now see something like this:</p><pre>The document&#39;s file name is a3fe4f2b-8ef1-4984-ac98-1b571171a6d6.docx</pre><p>Glorious. Let’s take a moment to reflect on what we just learned:</p><ul><li>We created a custom activity that receives the document ID as an input and provides the loaded document as well as a file stream as an output.</li><li>We could have hardcoded the document ID value, but instead we provided input in the form of a JavaScript expression.</li><li>We then connected the HTTP Response activity to write back the loaded document’s file name by leveraging Liquid.</li><li>We learned that we had to configure Liquid and allow-list our custom types if we want to access objects of these types.</li><li>We used the Elsa API to execute our workflow.</li></ul><p>Next, let’s work on the <strong>Archive Document</strong> activity.</p><h3>Archive Document</h3><p>The <strong>Archive Document</strong> activity is a very simple one with only two tasks it needs to perform:</p><ol><li>Set the <code>Status</code> property of a given <code>Document</code> to <code>Archived</code>.</li><li>Update the document in the database.</li></ol><p>The activity will take one input, which is the <code>Document</code> to update.</p><p>The complete file, <code>ArchiveDocument.cs</code>, to create looks like this:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=ArchiveDocument.cs.js"></script></figure><p>Although this activity is quite boring, it does showcase how easy it can be to quickly implement a custom activity that does something useful.</p><p>To try out this activity, connect it to the Get Document activity of the workflow we created earlier and use the following JS expression for its <code>Document</code> property:</p><pre>input.Document</pre><p>This works because the output of the Get Document activity returns an object of type <code>DocumentFile</code>, which contains a <code>Document</code> and a <code>FileStream</code> property. Since the <strong>Archive Document</strong> activity executes immediately after the <strong>Get Document</strong> activity, the output of the former will be available as the input to the latter.</p><p>However, there’s a big caveat: the HTTP Response activity is now no longer connected directly to the Get Document activity, so it can no longer use the <code>Input</code> variable.</p><p>To fix this, we need to give the <strong>Get Document</strong> activity a name. Giving activities a name allows you to access these activities’ output from any other activity in the workflow.</p><p>So here’s what we will do:</p><ol><li>Give the <strong>Get Document</strong> a name of <code>GetDocument1</code>.</li><li>Update the <strong>HTTP Response</strong> activity with the following liquid content:</li></ol><pre>The document&#39;s file name is {{ Activities.GetDocument1.Output.Document.FileName }} and the document&#39;s status is {{ Activities.GetDocument1.Output.Document.Status }}</pre><p>The workflow should look like this:</p><figure><img data-image-id="1*ZEPlfymt20ud2shXAHtbmA.png" data-width="970" data-height="1317" src="https://cdn-images-1.medium.com/max/800/1*ZEPlfymt20ud2shXAHtbmA.png"></figure><p>The response should be something like:</p><pre>The document&#39;s file name is b01bd191-2891-44d2-84e5-c8d4bfbb6917.docx and the document&#39;s status is 1</pre><p>Let’s look at a more interesting activity next.</p><h3>Zip File</h3><p>This activity is more fun because it does something I’m not doing everyday. The sole purpose of this one is to take a stream and returning a compressed version of it.</p><p>It takes two input properties:</p><ol><li>The stream to compress.</li><li>The filename to use when writing an entry to the zip archive.</li></ol><p>Here’s the code to create:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=ZipFile.cs.js"></script></figure><p>Nothing really special going on, but the application is awesome. Try it out by updating the workflow and connect the <strong>Zip File</strong> activity to the <strong>Archive Document</strong> using the following settings:</p><ul><li>Stream (JS): <code>activities.GetDocument1.Output().FileStream</code></li><li>File Name (JS): <code>activities.GetDocument1.Output().Document.FileName</code></li></ul><p>Then update the <strong>HTTP Response</strong> to write back the zipped stream as a download by configuring it with the following settings:</p><ul><li>Content (JS): <code>input</code></li><li>Content Type (Literal): <code>application/zip</code></li><li>Response Headers (Advanced tab): <code>{ \&quot;Content-Disposition\&quot;: \&quot;attachment; filename=download.zip\&quot; }</code></li></ul><p>The complete workflow should look like this:</p><figure><img data-image-id="1*hIST1SkvlcO19qKo8Z5pgQ.png" data-width="746" data-height="1296" src="https://cdn-images-1.medium.com/max/800/1*hIST1SkvlcO19qKo8Z5pgQ.png"></figure><p>When you now upload a document and execute the workflow using Postman, you should receive binary content. Use Postman to save the response as a zip file to make sure you can open the file and checkout the contents.</p><figure><img data-image-id="1*lzeROeIV76H6ro5tUGJ8Jw.png" data-width="607" data-height="254" src="https://cdn-images-1.medium.com/max/800/1*lzeROeIV76H6ro5tUGJ8Jw.png"></figure><p>If you think this is cool, then I agree!</p><p>Let’s move on to the final activity to implement: <strong>Update Blockchain</strong>.</p><h3>Update Blockchain</h3><p>Although the name of the activity might be very exciting, we aren’t actually writing anything to any blockchain. It‘s not just a ruse either; I can totally imagine someone writing an activity that writes a document signature to a blockchain as part of a document management system to make it tamper proof.</p><p>The more important point here is this: there may be activities that are better run in the background while the workflow goes to sleep and awaits completion of the background task.</p><p>So that’s what we will be showcasing next: writing an activity that starts a task, suspends the workflow while the task is running, and then let the task resume the workflow once it’s finished.</p><p>For this, we will leverage Hangfire as the engine for running a task in the background.</p><p>As a first step, let’s define the activity skeleton like this:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=UpdateBlockchain-1.cs.js"></script></figure><p>The activity receives as an input a <code>File</code>. This can either by a <code>Stream</code> or a <code>byte[]</code>. It then computes a hash and suspends the workflow.</p><p>At this point, there’s nothing that will ever resume this activity. Let’s take care of that next.</p><p>First, let’s inject Hangfire’s <code>IBackgroundJobClient</code> service and define a new <code>record</code> type that will carry some context for the background job to work on:</p><pre>public record UpdateBlockchainContext(string WorkflowInstanceId, string ActivityId, string FileSignature);</pre><p>Then replace the <code>TODO</code> comment in <code>OnExecuteAsync</code> with the following line:</p><pre>// Schedule background work using Hangfire.<br>_backgroundJobClient.Enqueue(() =&gt; SubmitToBlockChainAsync(new UpdateBlockchainContext(context.WorkflowInstance.Id, context.ActivityId, fileSignature), CancellationToken.None));</pre><p>This will enqueue a new job that is implemented in the form of the <code>SubmitToBlockChainAsync</code> method, which we’ll define next (as a public method within the activity class) as:</p><pre>/// &lt;summary&gt;<br>/// Invoked by Hangfire as a background job.<br>/// &lt;/summary&gt;<br>public async Task SubmitToBlockChainAsync(UpdateBlockchainContext context, CancellationToken cancellationToken)<br>{<br>    // Simulate storing it on an imaginary blockchain out there.<br>    await Task.Delay(TimeSpan.FromSeconds(15), cancellationToken);<br>}</pre><p>Once the job is done (waiting for 15 seconds), we need to resume the workflow. A good way to do is is to leverage <code>IWorkflowInstanceDispatcher</code>, which lets us schedule an existing workflow instance by ID for execution.</p><p>Go ahead and inject <code>IWorkflowInstanceDispatcher</code> and then add the following line to the <code>SubmitToBlockChainAsync</code> method:</p><pre>// Resume the suspended workflow.<br>await _workflowInstanceDispatcher.DispatchAsync(new ExecuteWorkflowInstanceRequest(context.WorkflowInstanceId, context.ActivityId, new WorkflowInput(context.FileSignature)), cancellationToken);</pre><p>Now you also see why our little <code>UpdateBlockchainContext</code> record type carries both the workflow instance ID as well as our activity ID — we need both these values to resume the workflow that will become blocked by our activity.</p><p>We also simply echo back the <code>FileSignature</code> by sending it as workflow input.</p><p>When the workflow is resumed, the workflow engine will invoke our activitiy’s <code>OnResumeAsync</code> method. Let’s update that now to receive the file signature and store it as output of our activity. Replace the <code>TODO</code> comment with the following:</p><pre>var fileSignature = context.GetInput&lt;string&gt;();<br>Output = fileSignature;</pre><p>The complete activity should look like this:</p><figure><script src="https://gist.github.com/sfmskywalker/cc4a4bbca1e805a5c43afc2af21b6af1?file=UpdateBlockchain-2.cs.js"></script></figure><h4>Try it</h4><p>Let’s try it out!</p><p>Update the workflow by connecting the <strong>Update Blockchain</strong> activity to the <strong>Zip File</strong> activity and configure it as follows:</p><ul><li>File (JS): <code>input</code></li></ul><p>Let’s also make sure we replace the <strong>HTTP Response</strong> activity, since the blockchain activity will put the workflow to sleep and resume it in the background where there’s no longer any HTTP context available.</p><p>So let’s do this:</p><ol><li>Delete the <strong>HTTP Response</strong> activity.</li><li>Connect a new Write Line activity with the following settings:</li></ol><ul><li>Text (Liquid): <code>File signature: {{ Input }}</code></li></ul><p>Publish the changes and invoke the workflow. Notice that the workflow will take about 15 seconds to complete now. Notice also that while you wait, the workflow is in the <em>Suspended</em> state:</p><figure><img data-image-id="1*jEFC6xnR9lmq7iPetTMhPg.png" data-width="1113" data-height="331" src="https://cdn-images-1.medium.com/max/800/1*jEFC6xnR9lmq7iPetTMhPg.png"></figure><p>After about 15 seconds, the console window should print something like the following output:</p><pre>File signature: ojPRr24fjiSUC0rQdVI4ieYoyy7Uhb1QfChtTKgStr4=</pre><h3>Next</h3><p>I don’t know about you, but I think writing custom activities is a lot of fun!</p><p>In the <a href="https://sipkeschoorstra.medium.com/part-7-of-building-workflow-driven-net-applications-with-elsa-2-e369e7784555" rel="noopener" target="_blank">next part</a>, we will be putting it all together and create a workflow for each document type and make good use of these custom activities, as well as some of the built-in activities.</p><p>See you there!</p></div></div></section>
