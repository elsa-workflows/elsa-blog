<section><div><div><h3>Part 5 of Building Workflow Driven .NET Applications with Elsa 2</h3><h4>Executing Workflows</h4><figure><img data-image-id="1*GDcj0rEH9P102mdrS9RaBQ.png" data-width="1444" data-height="619" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*GDcj0rEH9P102mdrS9RaBQ.png"></figure><p>In the <a href="https://sipkeschoorstra.medium.com/part-4-of-building-workflow-driven-net-applications-with-elsa-2-d6699a6a247e" rel="noopener" target="_blank">previous part</a>, we did the grunt work of setting up a basic ASP.NET Core web app project with UI, domain models and data access. We can now upload documents and store information in a database.</p><p>In this part, we will see how we can invoke workflows in response to uploaded files. In doing so, we can do all sorts of interesting things, such as sending the file for review as an email attachment, computing a hash, archive it, and whatever else floats your boat.</p><p>All of these things can be done without a workflows engine of course, but using a workflow engine offers a number of key advantages such as:</p><ul><li>Coordinating automated steps with human interaction becomes as simple as clicking together a single workflow. The key advantage here being that the state of this long-running process is handled for you.</li><li>Adding, updating and removing processes can be done without needing to redeploy the application (if you wanted to — you can still make workflows part of your application as we will see).</li><li>The developer provides the building blocks (workflow steps, a.k.a. activities) enabling non-programmers to create and manage workflows.</li></ul><h3>Executing Workflows</h3><p>Elsa provides various services to execute workflows, but at its core lies the <code>IWorkflowRunner</code> which takes a workflow blueprint and/or a workflow instance and the executes each activity one-by-one until either there are no more activities to be executed or until a blocking activity is encountered.</p><p>For our scenario, we want to execute workflows asynchronously in the background when a file is uploaded. To execute workflows in the background (using a queue worker for example), we use the <code>IWorkflowDispatcher</code> service.</p><p>This service will schedule a message on a queue (which is in-memory by default and uses <a href="https://github.com/rebus-org/Rebus" rel="noopener" target="_blank">Rebus</a>), which is then picked up by a message consumer, which in turn uses the <code>IWorkflowRunner</code> service to actually execute the workflow.</p><p>To see how this works, let’s first create a simple do-nothing workflow and then invoke this workflow in response to a file upload.</p><h4>Create HelloFile Workflow</h4><p>With your application and Elsa Studio running, create a workflow named <code>HelloFile</code> and add a single <strong>Write Line</strong> activity with the following Liquid expression:</p><pre>Hello File {{ Input }}! </pre><p><code>Input</code> will reference the object that we will sent as an input when invoking the workflow. Publish your workflow and let’s take a look at invoking it next.</p><h4>Run HelloFile Workflow</h4><p>As you may remember, in the previous part we created a service called <code>DocumentService</code> that will publish the <code>NewDocumentReceived</code> event anytime its <code>SaveDocumentAsync</code> is invoked (which is done from the <code>IndexModel</code> class in the web project upon uploading a file).</p><p>Now we are about to see how convenient that setup is. To handle the event, create a new folder called <code>Handlers</code> in the <code>DocumentManagement.Workflows</code> project and add to it the following class:</p><figure><script src="https://gist.github.com/sfmskywalker/ab8abe1021e6632b5bcff99116ad6e23?file=StartHelloFileWorkflow.cs.js"></script></figure><p>The handler shown above does two things:</p><ol><li>Get a <em>published</em> workflow blueprint by <em>name</em>.</li><li><em>Dispatch</em> the workflow.</li></ol><p>Before this handler is recognized, we need to register it with the service container. To do so, open the <code>ServiceCollectionExtensions</code> class in the same project and chain the following line in the <code>AddWorkflowServices</code> method:</p><pre>services.AddNotificationHandlersFrom&lt;StartHelloFileWorkflow&gt;()</pre><p>This will register any &amp; all notification handlers found in the current assembly.</p><h4>Test HelloFile Workflow</h4><p>To make sure everything works, start your web app and upload a file. After a short while (anywhere between 0 and 15 seconds), you should see something like the following output appear in the console window:</p><figure><img data-image-id="1*Jq0v11Q2sdITauImuWEQxQ.png" data-width="467" data-height="117" src="https://cdn-images-1.medium.com/max/800/1*Jq0v11Q2sdITauImuWEQxQ.png"></figure><p>Great! We now know how to run workflows whenever a document is uploaded.</p><p>Next, let’s see how we can select workflows for a given document type.</p><h3>Workflows By Document Type</h3><p>If we come up with a mechanism that allows us to associate a document type with a workflow, our application will allow users to add arbitrary document types and associated workflows to handle these types without us having to make any changes (unless of course the user wants to implement a workflow with new capabilities for which we don’t have an activity yet).</p><p>An easy way to associate workflows with document types is by <em>tagging</em> the workflow.</p><p>For example, we have a document type with ID <code>&quot;ChangeRequest&quot;</code>. Let’s update or <strong>HelloFile</strong> workflow’s settings and enter this type ID in its <em>Tag</em> field (under the <em>Advanced</em> tab):</p><figure><img data-image-id="1*FzhSaTuyg7aKKxJMDsqZdw.png" data-width="924" data-height="613" src="https://cdn-images-1.medium.com/max/800/1*FzhSaTuyg7aKKxJMDsqZdw.png"></figure><p>Next, let’s update the our event handler class as follows:</p><ul><li>Rename to <code>StartDocumentWorkflows</code>.</li><li>Select ALL workflows where their tag matches the document type ID of the uploaded document.</li><li>Instead of sending the document ID as workflow input, let’s use the document ID as the correlation ID for the workflow. This will be useful when implementing long-running workflows as we will see in the next part of this series.</li></ul><p>The final result should look like this:</p><figure><script src="https://gist.github.com/sfmskywalker/ab8abe1021e6632b5bcff99116ad6e23?file=StartDocumentWorkflows.cs.js"></script></figure><p>Let’s also update the <em>HelloFile</em> workflow’s <em>Write Line</em> activity to print out the correlation ID instead of the workflow input (which we no longer send):</p><pre>Hello File {{ CorrelationId }}!</pre><p>When you now try uploading documents, notice that the <em>HelloFile</em> workflow will only be executed when you upload documents of type <em>ChangeRequest</em>.</p><h3>Summary</h3><p>In this part, we’ve seen how to execute workflows in the background in response to a domain event (<code>NewDocumentReceived</code>) and how to associate workflows with document types via the <em>Tag</em> attribute.</p><h3>Next</h3><p>In the <a href="https://sipkeschoorstra.medium.com/part-6-of-building-workflow-driven-net-applications-with-elsa-2-9b3167c612dd" rel="noopener" target="_blank">next part</a>, we will take a look at building out a few custom activities which we need to implement the workflows described in the introduction of this series.</p></div></div></section>
