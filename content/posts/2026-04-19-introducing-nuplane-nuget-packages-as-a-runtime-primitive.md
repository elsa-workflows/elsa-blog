<figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*G0FKt3R3fyuYfxEoHshuHQ.jpeg" /><figcaption>NuGet Packages as a Runtime Primitive</figcaption></figure><p>Most .NET applications treat NuGet packages as a build-time concept. You add a reference, run dotnet restore, compile, and ship. The packages are baked into the binary. If you want to update them, you redeploy.</p><p>That works fine for most cases. But there’s a class of problems where it doesn’t, where you actually need to extend a running application without taking it down. Hot-reload plugin systems. SaaS platforms that let tenants customize behavior. Workflow engines where operators need to deploy new steps without a maintenance window. Internal tool hosts that should just pick up new capabilities as they’re pushed.</p><p>For those cases, the usual pattern is to roll your own packaging and loading system: copy some DLLs to a folder, write some AssemblyLoadContext plumbing, build a file watcher, figure out rollback, and add enough logging to understand what happened when things go wrong. It&#39;s not impossible, but it&#39;s a lot of infrastructure to carry for what is fundamentally a solved problem.</p><p><a href="https://nuplane.io/">Nuplane</a> is a .NET library that handles this infrastructure. It treats NuGet packages as a runtime primitive. Something your application can install, update, and load while it’s running.</p><h3>What Nuplane Actually Does</h3><p>At its core, Nuplane implements a deterministic control loop:</p><ol><li>Determine the <strong>desired package set</strong> (from a local folder, a NuGet feed, or both)</li><li>Compare against <strong>current state</strong> (what’s already installed and active)</li><li>Compute the diff</li><li>Apply transactional per-package changes to a local store</li><li>Optionally load the resulting assemblies into isolated contexts</li><li>Emit change events your host can react to</li></ol><p>That loop runs on a configurable poll interval, or fires immediately when a file watcher detects a new .nupkg in a watched directory. The result is that you can drop a package file into a folder and have its types available in the running process within about a second.</p><p>A few properties make this more than just a wrapper around AssemblyLoadContext:</p><p><strong>Deterministic storage.</strong> Packages live in a predictable on-disk structure. Nuplane manages active version pointers, staging, and prior versions independently. Updates are atomic: download to staging, validate, move to the immutable store, switch the active pointer, persist state. If anything in that chain fails, the previous version stays active.</p><p><strong>Last-known-good (LKG) protection.</strong> When an update fails, e.g., due to a bad package, missing dependency, load error, etc., Nuplane preserves the last successfully applied version rather than leaving the system in a half-updated state. The package stays active; the failure is logged and surfaced through health checks.</p><p><strong>Host neutrality.</strong> Nuplane does not define plugin semantics, mutate your DI container, or tell your application what to do with loaded types. It resolves and installs packages, loads assemblies, and fires events. What you call those packages, be it plugins, feature modules, extensions, rule sets, and how you activate and configure them is entirely your problem. That’s a deliberate boundary.</p><h3>The Drop-Folder Workflow</h3><p>The fastest way to understand Nuplane is through its drop-folder behavior. You configure a directory as a feed, enable the file watcher, and add the optional loading module:</p><pre>var builder = WebApplication.CreateBuilder(args);<br>var nuplaneConfiguration = builder.Configuration.GetSection(&quot;Nuplane&quot;);<br><br>builder.Services.AddNuplane(nuplaneConfiguration, nuplane =&gt;<br>{<br>    nuplane.AddDirectoryFeedsFromConfiguration(nuplaneConfiguration);<br>    nuplane.AutoloadPackages(nuplaneConfiguration.GetSection(&quot;Loading&quot;));<br>    nuplane.OnPackagesChanged&lt;PackageChangeObserver&gt;();<br>    nuplane.OnPackagesChanged&lt;PluginDiscoveryObserver&gt;();<br>});</pre><p>The configuration side of this declares the folder feed, the watcher behavior, and loading settings:</p><pre>{<br>  &quot;Nuplane&quot;: {<br>    &quot;Setup&quot;: {<br>      &quot;AutomaticReconciliation&quot;: true,<br>      &quot;PollInterval&quot;: &quot;00:01:00&quot;,<br>      &quot;Feeds&quot;: [<br>        {<br>          &quot;Name&quot;: &quot;local-packages&quot;,<br>          &quot;DirectoryPath&quot;: &quot;packages&quot;,<br>          &quot;IncludePatterns&quot;: [&quot;*&quot;],<br>          &quot;Directory&quot;: {<br>            &quot;Watch&quot;: true,<br>            &quot;DebounceWindow&quot;: &quot;00:00:01&quot;<br>          }<br>        }<br>      ]<br>    },<br>    &quot;Loading&quot;: {<br>      &quot;Enabled&quot;: true,<br>      &quot;SharedAssemblies&quot;: [<br>        {<br>          &quot;Name&quot;: &quot;MyApp.Abstractions&quot;,<br>          &quot;PublicKeyToken&quot;: &quot;31bf3856ad364e35&quot;,<br>          &quot;MajorVersion&quot;: 1<br>        }<br>      ]<br>    }<br>  }<br>}</pre><p>The SharedAssemblies list is how you tell Nuplane which contracts (your plugin interfaces, your shared model types) should be resolved from the host rather than loaded fresh per-package. This is the standard mechanism for making types from different isolated load contexts assignable to each other, which is the usual stumbling block when you first roll your own plugin loading.</p><p>Once that’s wired up, the workflow is:</p><pre># Build the plugin package<br>dotnet pack MyPlugin/MyPlugin.csproj -c Release<br><br># Drop it into the watched folder<br>cp MyPlugin/bin/Release/MyPlugin.1.0.0.nupkg packages/</pre><p>Nuplane’s watcher fires (with a 1-second debounce), reconciliation runs, assemblies load, and your registered observers are called.</p><h3>Reacting to Package Changes</h3><p>Observers in Nuplane are simple: implement INuplaneObserver and register it with OnPackagesChanged&lt;T&gt;(). The interface gives you callbacks for the start of a change, the completion, and per-package failures.</p><pre>internal sealed class PackageChangeObserver(ILogger&lt;PackageChangeObserver&gt; logger)<br>    : INuplaneObserver<br>{<br>    public Task OnPackagesChangingAsync(PackageChangeSet changeSet, CancellationToken ct)<br>    {<br>        logger.LogInformation(<br>            &quot;Packages changing. Added={AddedCount}, Updated={UpdatedCount}, CorrelationId={CorrelationId}&quot;,<br>            changeSet.Added.Count,<br>            changeSet.Updated.Count,<br>            changeSet.CorrelationId);<br> <br>       return Task.CompletedTask;<br>    }<br><br>    public Task OnPackagesChangedAsync(PackageChangeSet changeSet, CancellationToken ct)<br>    {<br>        logger.LogInformation(<br>            &quot;Packages changed. Added={AddedCount}, Updated={UpdatedCount}, Removed={RemovedCount}&quot;,<br>            changeSet.Added.Count,<br>            changeSet.Updated.Count,<br>            changeSet.Removed.Count);<br><br>       return Task.CompletedTask;<br>    }<br><br>    public Task OnPackageFailedAsync(string packageId, Exception exception, CancellationToken ct)<br>    {<br>        logger.LogWarning(exception, &quot;Package operation failed for {PackageId}&quot;, packageId);<br>        return Task.CompletedTask;<br>    }<br>}</pre><p>One important thing about observers: they’re invalidation signals, not the authoritative source of state. The recommended pattern is to treat an observer callback as a trigger to re-query the catalog, not as an event to replay or accumulate. This matters in practice because it keeps your host from building a fragile event-history model that diverges from actual runtime state.</p><h3>Reading State from the Catalog</h3><p>When you need the current state of loaded packages and types, Nuplane exposes a set of query surfaces rather than expecting you to track state through observer callbacks.</p><p>The loading module provides two surfaces you’ll use frequently:</p><p>IPackageAssemblyCatalog: the primary surface for loading-enabled hosts. It gives you the currently active loaded assemblies across all active packages, or for a specific package ID.</p><p>IPackageTypeFinder: convenience wrapper over the assembly catalog that applies assignability filtering. Hand it an interface type and a package ID and it returns all implementations currently loaded from that package.</p><p>A typical host-owned plugin discovery service looks like this:</p><pre>internal sealed class PluginCatalog(<br>    IPackageAssemblyCatalog packageAssemblyCatalog,<br>    IPackageTypeFinder packageTypeFinder)<br>{<br>    public async Task&lt;IReadOnlyList&lt;DiscoveredPlugin&gt;&gt; DiscoverAsync(CancellationToken ct)<br>    {<br>        var discovered = new List&lt;DiscoveredPlugin&gt;();<br><br>        foreach (var package in (await packageAssemblyCatalog.GetPackagedAssembliesAsync(ct))<br>                     .Where(p =&gt; p.AssemblyReferences.Count &gt; 0))<br>        {<br>            var pluginTypes = await packageTypeFinder.FindTypesAsync(<br>                typeof(IPlugin), package.PackageId, ct);<br><br>            foreach (var pluginType in pluginTypes)<br>            {<br>                discovered.Add(new DiscoveredPlugin(<br>                    package.PackageId,<br>                    package.Version,<br>                    pluginType));<br>            }<br>        }<br>        return discovered;<br>    }<br>}</pre><p>That’s the full pattern: Nuplane handles the package lifecycle, you query what’s loaded, and you do whatever your application needs with the resulting types. The assembly loading boundary and the shared assembly configuration handle the cross-context type compatibility for you.</p><p>If you don’t need full loading and only want package inventory visibility (maybe you’re tracking which versions are active for observability, or coordinating a package state across services) you can use the core runtime without the loading module at all.</p><h3>Remote Feed Support</h3><p>The drop-folder workflow is the easiest to demonstrate, but Nuplane also supports NuGet v3 remote feeds with polling:</p><pre>builder.Services.AddNuplane(nuplane =&gt;<br>{<br>    nuplane.PollEvery(TimeSpan.FromMinutes(1));<br>    nuplane.AddFeed(&quot;nuget.org&quot;, feed =&gt;<br>    {<br>        feed.FromUri(&quot;https://api.nuget.org/v3/index.json&quot;);<br>        feed.Include(&quot;MyCompany.Plugins.*&quot;);<br>    });<br>});</pre><p>Or declaratively in configuration, using the full Nuplane:Setup:Feeds shape and pointing at nuget.org or an internal feed with an include pattern to filter which packages Nuplane tracks:</p><pre>{<br>  &quot;Nuplane&quot;: {<br>    &quot;Setup&quot;: {<br>      &quot;Feeds&quot;: [<br>        {<br>          &quot;Name&quot;: &quot;nuget.org&quot;,<br>          &quot;ServiceIndex&quot;: &quot;https://api.nuget.org/v3/index.json&quot;,<br>          &quot;IncludePatterns&quot;: [&quot;MyCompany.Plugins.*&quot;]<br>        }<br>      ]<br>    }<br>  }<br>}</pre><p>One thing worth noting about include patterns: a feed with no patterns defined contributes nothing. This is intentional. The default is restrictive ; you have to be explicit about what a feed is authoritative for. This matters when you’re mixing internal and public feeds and you don’t want accidental resolution crossover.</p><h3>Where Nuplane Fits</h3><p>The cases where Nuplane earns its keep tend to share a few properties: the host is long-running, extensions come from sources outside the host’s own deployment, and the cost of a restart is real.</p><p><strong>Hot-reload plugin systems</strong>. This is the primary use case the sample demonstrates. Package your plugins as .nupkg files, point Nuplane at a watched folder, and you have a working plugin loading system with transactional safety and LKG fallback built in.</p><p><strong>Modular feature delivery in SaaS platforms</strong>. If your application is split into independently versioned feature packages, Nuplane lets you update a single feature at runtime. The others keep running. This is useful when different parts of the product are on different release cadences and you need fine-grained control over what’s live.</p><p><strong>Per-tenant customization</strong>. Load per-tenant behavior packages dynamically, isolated per load context. Each tenant’s package set is independent; a bad package from one tenant doesn’t affect others. The isolation model here is the same AssemblyLoadContext isolation .NET provides; Nuplane wires it up and manages the shared contract boundaries.</p><p><strong>Workflow and rule engines</strong>. Operators need to deploy new steps, validators, or routing rules without a maintenance window. With Nuplane, those are packages. Push a new version; the engine picks it up live.</p><p><strong>Internal platform hosting</strong>. Push a new package to a shared internal folder; every host watching that folder reconciles automatically. No CI/CD pipeline change, no coordinated deployment.</p><h3>What to Watch Out For</h3><p>Nuplane is infrastructure, and like most infrastructure it requires you to think about the edges.</p><p>Assembly unload is not guaranteed. .NET’s AssemblyLoadContext unload is cooperative and can fail if live references remain. Nuplane tracks unload-pending state and retries, but if your host holds references to types from an old assembly version, you can end up with multiple versions loaded simultaneously. Design your plugin activation patterns accordingly.</p><p>It does not sandbox untrusted code. If you’re loading packages from external or user-provided sources, enforcement of what that code can do is your responsibility. Nuplane has hooks for package validation (hash checks, signature checks, allowlists), but it does not run loaded code in a restricted context.</p><p>Kubernetes needs persistent storage. Nuplane keeps its package store and state file on disk. If a pod restarts with an empty filesystem, it will re-download and re-extract packages from scratch. For production deployments on Kubernetes, mount a persistent volume for both the package install root and the state file. A StatefulSet with per-replica volumes is the recommended shape.</p><p>Omitted include patterns mean no packages. If you configure a feed without IncludePatterns or IncludeAll, that feed contributes nothing.</p><h3>The Module Layout</h3><p>Nuplane is split into focused packages so you only take what you need:</p><iframe src="https://cdn.embedly.com/widgets/media.html?src=https%3A%2F%2Fdatawrapper.dwcdn.net%2FkzlpQ%2F1%2F&amp;display_name=Datawrapper&amp;url=https%3A%2F%2Fdatawrapper.dwcdn.net%2FkzlpQ%2F1%2F&amp;image=https%3A%2F%2Fdatawrapper.dwcdn.net%2FkzlpQ%2Fplain-s.png%3Fv%3D1&amp;type=text%2Fhtml&amp;schema=dwcdn" width="600" height="389" frameborder="0" scrolling="no"><a href="https://medium.com/media/ce4d899156e6630ea5da8fb88cf9a757/href">https://medium.com/media/ce4d899156e6630ea5da8fb88cf9a757/href</a></iframe><p>The loading module is genuinely optional; if all you need is package reconciliation and state tracking without loading assemblies into your process, the baseline modules work without it.</p><h3>Trying It Out</h3><p>The repository includes a working ASP.NET Core sample that demonstrates the full drop-folder to type discovery loop. The setup is:</p><pre># Start the host<br>dotnet run --project samples/Nuplane.Sample.AspNetCore/Nuplane.Sample.AspNetCore.csproj<br><br># In another terminal, pack the sample plugin<br>dotnet pack samples/Nuplane.Sample.Plugin/Nuplane.Sample.Plugin.csproj -c Debug<br><br># Drop it into the watched folder<br>mkdir -p packages<br>cp samples/Nuplane.Sample.Plugin/bin/Debug/Nuplane.Sample.Plugin.1.0.0.nupkg packages/</pre><p>Then query /catalog/plugins. The HelloPlugin type shows up in the response, discovered live in the running process with no restart. The whole loop from file drop to discoverable type takes about a second.</p><p>Nuplane is a focused piece of infrastructure for a problem that comes up more often than .NET’s standard tooling addresses. If you’re building something that needs NuGet packages at runtime such ss plugins, tenant customizations, live-updateable feature modules, then it’s worth a look.</p><p>The repository is at <a href="https://github.com/valence-works/nuplane">github.com/valence-works/nuplane</a>.</p>
