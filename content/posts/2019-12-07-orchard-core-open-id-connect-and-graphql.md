<section><div><div><h3>Orchard Core, Open ID and GraphQL</h3><h4>A first look at how to invoke Orchard Core’s GraphQL API endpoint using Postman</h4></div><div><figure><img data-image-id="1*24_Rpx6YmD6uZUpNrliGCw.jpeg" data-width="5184" data-height="3456" data-is-featured="true" src="https://cdn-images-1.medium.com/max/2560/1*24_Rpx6YmD6uZUpNrliGCw.jpeg"><figcaption>Photo by Caspar Camille Rubin on Unsplash</figcaption></figure></div><div><p>As I’m easing back into Orchard Core development, I’m running into little learning curves here and there. It’s been a while since I’ve been actively involved with the Orchard Core project, so it’s natural to be a little bit out of touch with how things work. It’s also natural for me to just dig in and figure it out and share my learnings.</p><p>So here’s a little learning about Orchard Core and GraphQL.</p><p>Enabling the GraphQL feature was easy. But then I had to figure out what URL to send my queries to. When I figured that out, I was getting a HTTP 401 Unauthorized response.</p><p>As it turns out, there are essentially two ways about this:</p><ol><li>Grant the Anonymous user role the <strong>Execute GraphQL</strong> permission, or;</li><li>Enable the <strong>Open ID Server</strong> feature and create an Open ID <em>Application</em> representing your client application. This application stores a <em>Client ID</em> and <em>Client Secret</em> that are used to request a <a href="https://jwt.io/" rel="noopener" target="_blank">Json Web Token</a>, JWT for short. This JWT is then used as a bearer token when sending GraphQL queries to the GraphQL endpoint.</li></ol><p>Let’s see how all of this works in more detail.</p></div></div></section><section><div><div><h3>Objective</h3><p>Here’s what we’ll achieve:</p><ol><li>Expose a GraphQL endpoint from Orchard to allow content querying.</li><li>Invoke the GraphQL endpoint from a client application. This could be plain vanilla JS, Angular, React, Blazor, and so on. I’m just gonna use Postman to try out the GraphQL endpoint. Once that works, we can do it from any other client application.</li></ol><h3>Orchard Setup</h3><p>I’m assuming you understand how to setup an Orchard Core CMS website. If not, I recommend you go through the <a href="https://orchardcore.readthedocs.io/en/dev/docs/getting-started/" rel="noopener" target="_blank">Getting started</a> tutorial first.</p><p>When you’re on the <em>Setup</em> screen, make sure to select the Blog recipe, as it will create the <em>BlogPost</em> content type and a sample blog post content item that we’ll use in the GraphQL query.</p></div></div></section><section><div><div><h3>Enabling the GraphQL Feature</h3><p>The first order of business is to enable the <strong>GraphQL</strong> feature. Go to the admin dashboard and look at the left menu. Select <em>Configuration -&gt; Features</em> and search for <em>GraphQL</em> and select the <em>Enable </em>button.</p><figure><img data-image-id="1*cLmWscVDcnL52akieHsMkQ.png" data-width="1911" data-height="1061" src="https://cdn-images-1.medium.com/max/800/1*cLmWscVDcnL52akieHsMkQ.png"><figcaption>Select “Enable” to enable the GraphQL feature.</figcaption></figure><p>The <em>Configuration</em> menu will now have an additional menu item called <em>GraphiQL</em>. <a href="https://github.com/graphql/graphiql" rel="noopener" target="_blank">GraphiQL</a> is an in-browser IDE for exploring GraphQL and enables you to try out queries. One of the most powerful features of GraphQL is its strongly-typed <a href="https://graphql.org/learn/schema/" rel="noopener" target="_blank">schemas</a>, which enables GraphiQL to provide autocomplete functionality as we explore the available types.</p><p>For example, if you wanted to query all blog posts, you could write the following query:</p><pre>query BlogPosts {<br>  blogPost {<br>    markdownBody {<br>      html<br>    }<br>  }<br>}</pre><p>Thanks to the schema and the GraphiQL UI, we don’t have to go through the source code to figure out all of the available types. Instead, we can simply select the available types and fields to build up our query:</p><figure><img data-image-id="1*XIN1fLb6Hjx8N5yLSm9Zjg.png" data-width="1423" data-height="1061" src="https://cdn-images-1.medium.com/max/800/1*XIN1fLb6Hjx8N5yLSm9Zjg.png"><figcaption>Use the Explorer to select fields to include in the query.</figcaption></figure><p>In addition to selecting fields to query, you can also leverage the <em>alt+enter</em> key combination in the query editor to invoke autocomplete:</p><figure><img data-image-id="1*8uK0yVM7wM1Rd83aPxmMwQ.png" data-width="554" data-height="774" src="https://cdn-images-1.medium.com/max/800/1*8uK0yVM7wM1Rd83aPxmMwQ.png"><figcaption>Press ALT+ENTER to invoke autocomplete.</figcaption></figure><p>When you hit the Play button. te query will be executed:</p><figure><img data-image-id="1*p5f5WbELh2XPEibR-28TTg.png" data-width="1390" data-height="1061" src="https://cdn-images-1.medium.com/max/800/1*p5f5WbELh2XPEibR-28TTg.png"><figcaption>Select “Play” to execute the query.</figcaption></figure><p>When I first saw this, it blew my mind. Being able to specify queries that return the exact amount of information I need for my SPA application is an exciting feat indeed! Compare this with having to maintain RESTful APIs that return DTOs tailored to clients. If you have one client, it’s OK. But once you start having both an Admin client and a front-end client for example, already things tend to become somewhat in-elegant. With GraphQL, I can simply expose a single API endpoint and allow clients to query what they need.</p><p>Now let’s see how we can invoke this endpoint using Postman.</p><h3>Invoking the GraphQL Endpoint using Postman</h3><p>Invoking the GraphQL endpoint can be done in a variety of ways, but first one must know its endpoint URL. By default, its the following path:</p><pre>/api/graphql</pre><p>The GraphQL endpoint can be invoked in the following ways:</p><ol><li>Using HTTP GET and specifying the query as a querystring value.</li><li>Using HTTP POST with <em>content-type</em> set to <em>application/json.</em></li><li>Using HTTP POST with <em>content-type</em> set to <em>application/graphql</em>.</li></ol><p>For more details on these options, make sure to checkout the <a href="https://orchardcore.readthedocs.io/en/dev/docs/reference/modules/Apis.GraphQL/" rel="noopener" target="_blank">Orchard Core GraphQL documentation</a>.</p><p>We’re going to give the HTTP POST / content-type = application/graphql option a try. Specifically, we will query all BlogPost content types.</p><p>The complete HTTP request looks something like this:</p><pre>POST /api/graphql HTTP/1.1<br>Host: localhost:8396<br>Content-Type: application/graphql</pre><pre>query {<br>   blogPost {<br>      markdownBody {<br>        html<br>      }<br>   }<br>}</pre><p>Here’s what it looks like in Postman:</p><figure><img data-image-id="1*7ioCClxuD-1xGU8qsG7z0w.png" data-width="1155" data-height="846" src="https://cdn-images-1.medium.com/max/800/1*7ioCClxuD-1xGU8qsG7z0w.png"><figcaption>Postman</figcaption></figure><p>Make sure to set the content-type to application/graphql, like so:</p><figure><img data-image-id="1*F2CD_r31EhoQjCWmDZbmxg.png" data-width="784" data-height="361" src="https://cdn-images-1.medium.com/max/800/1*F2CD_r31EhoQjCWmDZbmxg.png"><figcaption>Content-Type is set to application/graphql</figcaption></figure><p>This looks good. However, when selecting the Send button, the response may not be what you might have expected. At least, I didn’t:</p><pre>HTTP 400 Bad Request</pre></div></div></section><section><div><div><p>While debugging to understand what the reason might be for the Bad Request, I noticed that in fact, the GraphQL Middleware component responded with a <strong>HTTP 401 Unauthorized</strong> status code. I’m not sure how that resulted in a HTTP 400 Bad Request status in Postman, but I figured that I should at least figure out how to deal with the Unauthorized response first.</p><p>There are two ways to deal with this, and the best way depends on your use case:</p><ol><li>Grant the <em>Execute GraphQL</em> permission to the Anonymous user role.</li><li>Create an Open ID Connect Application and use its Client ID and Client Secret to request an access token (a JWT). Use this access token when making GraphQL requests.</li></ol><p>We’ll take a look at both options. First, we’ll try option 1:</p><h3>Granting the <em>Execute GraphQL</em> permission to the Anonymous user role</h3><p>If your site content is allowed to be publicly available and you have for example a static a front-end application built with Angular, then the best option might be to just allow anyone to make GraphQL requests. This allows your SPA application to invoke the GraphQL endpoint directly, without having the request to be proxied through a back-end controller or middleware.</p><p>To allow for this, follow these steps:</p><ol><li>Select the <em>Security -&gt; Roles</em> menu item from the admin menu.</li><li>Select the <em>Edit</em> button for the <em>Anonymous</em> role.</li><li>Search for the “Execute GraphQL” permission and check its “Allow” checkbox.</li><li>Select the <em>Save</em> button at the very bottom of the screen.</li></ol><p>Now try to execute the HTTP request from Postman again. This time, we see success:</p><figure><img data-image-id="1*PijIkILfP_BWhojdEeHqXQ.png" data-width="777" data-height="629" src="https://cdn-images-1.medium.com/max/800/1*PijIkILfP_BWhojdEeHqXQ.png"><figcaption>A successful GraphQL response.</figcaption></figure><h3>Making Authenticated GraphQL Calls</h3><p>In order to make authenticated calls, the caller needs to have the <strong>ExecuteGraphQL</strong> permission. In the previous example, we assigned this permission to the Anonymous role, effectively granting this permission to anonymous users.</p><p>But this might not always be desirable. For example, you may want to control what applications are allowed to execute GraphQL queries.</p><p>Fortunately, Orchard makes this again very easy for us. As it turns out, the <a href="https://orchardcore.readthedocs.io/en/dev/docs/reference/modules/OpenId/" rel="noopener" target="_blank">Open ID module</a> provides a feature that allows us to create an OAuth2 application, which enables all of the OAuth2 grant types we want to support.</p><blockquote>A <strong>Grant Type</strong> is also known as a <strong>Flow</strong>.</blockquote><p>In this example, we’ll use the OAuth 2.0 Client Credentials Grant type.</p><p>First, make sure to remove the <em>Execute GraphQL</em> permission from the Anonymous role.</p><p>Then enable the following two features:</p><ul><li><strong>OpenID Authorization Server</strong></li><li><strong>OpenID Token Validation</strong></li></ul><figure><img data-image-id="1*wHmzai1a_aUAftszaR2A5g.png" data-width="1168" data-height="760" src="https://cdn-images-1.medium.com/max/800/1*wHmzai1a_aUAftszaR2A5g.png"></figure><blockquote>The OpenID token Validation feature enables Bearer token validation.</blockquote><p>With these features enabled, select <em>Security → OpenID Connect → Settings → Authorization Server</em> from the admin menu and check the <strong>Allow Client Credentials Flow</strong>. Make sure to select the <em>Update the server settings and reload the tenant </em>button at the very bottom of the screen.</p><figure><img data-image-id="1*2rYHkZB2OORw5mMAwIC8Mg.png" data-width="1168" data-height="760" src="https://cdn-images-1.medium.com/max/800/1*2rYHkZB2OORw5mMAwIC8Mg.png"><figcaption>OpenID Connect Authorization Server settings.</figcaption></figure><p>Next, select the <em>OpenID Connect → Management → Applications</em> menu item from the admin menu and select the <em>Add an application</em> button.</p><figure><img data-image-id="1*PVaj2IA_OGkC8f1kG-clCg.png" data-width="1168" data-height="760" src="https://cdn-images-1.medium.com/max/800/1*PVaj2IA_OGkC8f1kG-clCg.png"></figure><p>You can provide any value you like in the <em>Client id</em>, <em>Display Name</em> and <em>Client Secret</em> fields. Just make sure to also check the <em>Allow Client Credentials Flow</em> checkbox. I entered the following values:</p><ul><li>Client id: <em>e0f660a2cf2a47babac40a4a8c24e7e0</em></li><li>Display Name: <em>Postman</em></li><li>Type: <em>Confidential client</em></li><li>Client Secret: <em>76945d3917a4456db5a41fc2949d6439</em></li><li>Client Credentials Roles: <em>Administrator</em></li></ul><blockquote>Make sure to take a note of the Client Secret, because it will not be displayed again.</blockquote><p>Select the <em>Save</em> button to create the application.</p><p>Notice that I selected the <strong>Administrator</strong> role for the application. Be aware that this grants your application administrative access to the Orchard Core tenant (which includes the <em>Execute GraphQL</em> permission, which is why I chose this role). A safer option would be to <em>create a separate role</em> that has only the <em>Execute GraphQL</em> permission applied.</p><h4>Requesting Access Tokens</h4><p>Now that we have an application representing our Postman client, we can use the client credentials to request an access token. The HTTP request looks like this:</p><pre>POST /connect/token HTTP/1.1<br>Host: localhost:8396<br>Content-Type: application/x-www-form-urlencoded</pre><pre>client_id=e0f660a2cf2a47babac40a4a8c24e7e0&amp;client_secret=76945d3917a4456db5a41fc2949d6439&amp;grant_type=client_credentials</pre><p>Invoking that request yields the following JSON response:</p><pre>{&quot;token_type&quot;:&quot;Bearer&quot;,&quot;access_token&quot;:&quot;CfDJ8GB7ya...trimmed...&quot;,&quot;expires_in&quot;:3600}</pre><p>Our access token is provided via the <code>access_token</code> field, which we use when creating GraphQL requests, such as the following:</p><pre>POST /api/graphql HTTP/1.1<br>Host: localhost:8396<br>Content-Type: application/graphql<br>Authorization: Bearer CfDJ8GB7ya...trimmed...</pre><pre>query {<br>   blogPost {<br>      markdownBody {<br>        html<br>      }<br>   }<br>}</pre><p>Now we get back a valid HTTP 200 response again.</p></div></div></section><section><div><div><h3>Summary</h3><p>In this article, we’ve seen how to enable &amp; invoke the Orchard Core GraphQL endpoint. In order to be able to invoke this endpoint the caller must have the <code>Execute GraphQL</code> permission assigned. We’ve seen how to assign this permission to a user role and how to create an OAuth2 Application and assign a role to it. We then used the application’s credentials to request an access token, which is sent as a bearer token alongside GraphQL requests.</p><p>Being able to query Orchards GraphQL endpoint is a powerful feature and makes it easy to use Orchard Core as a headless CMS. Clients can simply request content that they need, and thanks to GraphQL’s flexibility, even shape the returned response.</p></div></div></section>
