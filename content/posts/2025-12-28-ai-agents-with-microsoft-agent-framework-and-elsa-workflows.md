<h4>Agentic workflows are becoming a first-class architectural concern.</h4><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*K1pV07McSDkDHbm_2bbLKg.png" /></figure><blockquote><strong>TL;DR</strong><br>We start with an agent-native workflow using Microsoft Agent Framework, encapsulate it as a reusable capability, and then move orchestration into Elsa Workflows. The result is a clear mental model for when agents should orchestrate, and when workflows should.</blockquote><p>I’ve recently started exploring Microsoft Agent Framework, a set of libraries for .NET developers who want to build advanced AI agents with clearly defined skills, orchestrated by so-called <em>workflow agents</em>. From what I’ve seen so far, it’s very impressive. These workflow agents coordinate multiple agents to collaborate toward a shared goal.</p><p>Adding cognition to your application has never been easier. And because agents can be invoked just like any other capability, the same now applies to your workflow applications.</p><p>All it takes is creating a custom activity and calling your agent from it. Even better, with the release of Elsa 3.6, .NET classes and their public methods can now be called directly from workflows.</p><p>Let’s see how that works.</p><h3>What we’re building</h3><p>To demonstrate how to create agents using Microsoft Agent Framework and then leverage them as activities in Elsa Workflows, we’ll work through a series of milestones. Each milestone represents an incremental architectural step, starting from a basic ASP.NET Core application and ending with a full-fledged Elsa Workflows engine orchestrating agents.</p><p>In the sample, we’ll define three agents:</p><ol><li><em>Writer Agent</em>: writes a story on a given topic in a specific genre</li><li><em>Editor Agent</em>: reviews the story, fixes grammar, and improves the plot</li><li><em>Workflow Agent</em>: orchestrates the Writer and Editor agents</li></ol><p>We’ll start from an empty ASP.NET Core application targeting .NET 10 and incrementally build toward:</p><ol><li>A multi-agent workflow using Microsoft Agent Framework (Writer → Editor)</li><li>A reusable StoryWriterAgent that encapsulates that collaboration</li><li>An Elsa activity that encapsulates the StoryWriterAgent.</li><li>A refactoring where Elsa becomes the primary orchestration engine and agents become first-class workflow activities</li></ol><h3>Starting point: an empty ASP.NET Core app</h3><p>We’ll create a new empty ASP.NET Core project to keep the architecture explicit and transparent.</p><pre>dotnet new web -n AgentElsaDemo -f net10.0<br>cd AgentElsaDemo</pre><p>At this point, the project contains little more than Program.cs.</p><h3>Milestone 1: Agent-native orchestration with Microsoft Agent Framework</h3><p>Microsoft Agent Framework includes a workflow model designed specifically for agent interactions:</p><ul><li><strong>Nodes</strong> represent agents</li><li><strong>Edges</strong> represent transitions</li><li><strong>Conditions</strong> control routing</li><li>The <em>workflow itself</em> can be executed as <em>a single agent</em></li></ul><p>In other words, it already <em>is</em> a workflow engine. Just a specialized one focussed on agent collaboration.</p><h4>Installing packages</h4><pre>dotnet add package Microsoft.Agents.AI --prerelease<br>dotnet add package Microsoft.Agents.AI.OpenAI --prerelease<br>dotnet add package Microsoft.Agents.AI.Workflows --prerelease<br>dotnet add Microsoft.SemanticKernel.Connectors.OpenAI</pre><p>These packages allow us to define agents backed by OpenAI models and compose them into agent-native workflows.</p><p>Configure your OpenAI API key via environment variables or configuration. In this sample, we’ll use OPENAI__APIKEY.</p><h4>Defining agents</h4><p>In Program.cs, we’ll define two simple agents:</p><ul><li><strong>Writer</strong>: produces a first draft</li><li><strong>Editor</strong>: improves that draft</li></ul><pre>var writer = chatClient.CreateAIAgent(<br>    name: &quot;Writer&quot;,<br>    instructions: &quot;Write a short story based on the provided topic.&quot;<br>);<br><br>var editor = chatClient.CreateAIAgent(<br>    name: &quot;Editor&quot;,<br>    instructions: &quot;Improve the draft: fix grammar, improve flow, and tighten the plot.&quot;<br>);</pre><h4>Building an agentic workflow</h4><p>Next, let’s create a sequential workflow that will invoke the Writer agent and Editor agent for us.</p><pre>var workflow = AgentWorkflowBuilder.BuildSequential(writer, editor);<br>var workflowAgent = workflow.AsAgent();<br><br>var result = await workflowAgent.RunAsync(<br>    &quot;Write a short story about a haunted lighthouse.&quot;<br>);</pre><p>From the outside, the entire Writer → Editor collaboration is exposed as a single callable agent. This is a powerful property: agent workflows can be composed hierarchically, just like regular agents, without leaking internal orchestration details.</p><p>What follows is the complete initial code listing, wired up as a minimal API endpoint.</p><pre>using Microsoft.Agents.AI.Workflows;<br>using Microsoft.Extensions.AI;<br><br>var builder = WebApplication.CreateBuilder(args);<br>var configuration = builder.Configuration;<br>var services = builder.Services;<br>var openApiKey = configuration[&quot;OpenAI:ApiKey&quot;]!;<br>services.AddOpenAIChatClient(&quot;gpt-4o&quot;, openApiKey);<br>var app = builder.Build();<br><br>app.MapGet(&quot;/&quot;, () =&gt; &quot;Hello World!&quot;);<br>app.MapPost(&quot;/write-story&quot;, async (IChatClient chatClient) =&gt;<br>{<br>    var writer = chatClient.CreateAIAgent(<br>        name: &quot;Writer&quot;,<br>        instructions: &quot;Write a short story based on the provided topic.&quot;<br>    );<br><br>    var editor = chatClient.CreateAIAgent(<br>        name: &quot;Editor&quot;,<br>        instructions: &quot;Improve the draft: fix grammar, improve flow, and tighten the plot.&quot;<br>    );<br><br>    var workflow = AgentWorkflowBuilder.BuildSequential(writer, editor);<br>    var workflowAgent = workflow.AsAgent();<br><br>    var result = await workflowAgent.RunAsync(<br>        &quot;Write a short story about a haunted lighthouse.&quot;<br>    );<br>    <br>    return result.Text;<br>});<br><br>app.Run();</pre><p>When you run this program and send a POST request to /write-story, you should receive a short story about a haunted lighthouse after a few seconds.</p><h3>Milestone 2: Encapsulating the workflow as StoryWriterAgent</h3><p>The hard-coded orchestration logic in Program.cs is useful as a proof of concept, but not something you’d want to reuse or test easily. Let’s encapsulate it in a dedicated class:</p><pre>public class StoryWriterAgent(IChatClient chatClient)<br>{<br>    public async Task&lt;string&gt; WriteStoryAsync(string topic, string genre, CancellationToken cancellationToken = default)<br>    {<br>        var writer = chatClient.CreateAIAgent(<br>            name: &quot;Writer&quot;,<br>            instructions: &quot;Write a short story based on the provided topic.&quot;<br>        );<br><br>        var editor = chatClient.CreateAIAgent(<br>            name: &quot;Editor&quot;,<br>            instructions: &quot;Improve the draft: fix grammar, improve flow, and tighten the plot.&quot;<br>        );<br>        var workflow = AgentWorkflowBuilder.BuildSequential(writer, editor);<br>        var workflowAgent = workflow.AsAgent();<br>        var result = await workflowAgent.RunAsync(<br>            $&quot;Write a short story about {topic} in the genre of {genre}.&quot;, <br>            cancellationToken: cancellationToken);<br>        return result.Text;<br>    }<br>}</pre><p>At this point, we have a clean, self-contained agentic capability that can be invoked from anywhere: controllers, background services, workflows, or tests.</p><p>Updating the API to use this class keeps Program.cs simple and declarative.</p><pre>var builder = WebApplication.CreateBuilder(args);<br>var configuration = builder.Configuration;<br>var services = builder.Services;<br>var openApiKey = configuration[&quot;OpenAI:ApiKey&quot;]!;<br><br>services.AddOpenAIChatClient(&quot;gpt-4o&quot;, openApiKey);<br>services.AddSingleton&lt;StoryWriterAgent&gt;();<br><br>var app = builder.Build();<br><br>app.MapGet(&quot;/&quot;, () =&gt; &quot;Hello World!&quot;);<br>app.MapPost(&quot;/write-story&quot;, async (StoryWriterAgent agent, CancellationToken cancellationToken) =&gt; <br>    await agent.WriteStoryAsync(&quot;A haunted lighthouse&quot;, &quot;thriller&quot;, cancellationToken));<br><br>app.Run();</pre><h3>Milestone 3: Treating an agentic workflow as a process step</h3><p>We now have a reusable agent. The next step is to place it inside a broader, long-running process that can be visualized, persisted, and externally driven.</p><p>Before we can invoke it from an Elsa workflow, we need to turn our application into an Elsa Workflows engine.</p><h4>Adding Elsa Server</h4><p>First, add the necessary packages.</p><pre>dotnet add package Elsa --version 3.6.0<br>dotnet add package Elsa.Persistence.EFCore.Sqlite --version 3.6.0<br>dotnet add package Elsa.Workflows.Api --version 3.6.0<br>dotnet add package Elsa.Workflows.Runtime.Distributed --version 3.6.0</pre><p>Then update Program.cs as follows:</p><pre>using Elsa.Extensions;<br>using Elsa.Persistence.EFCore.Extensions;<br>using Elsa.Persistence.EFCore.Modules.Management;<br>using Elsa.Persistence.EFCore.Modules.Runtime;<br>using Elsa.Workflows.Runtime.Distributed.Extensions;<br>using WebApplication1;<br><br>var builder = WebApplication.CreateBuilder(args);<br>var configuration = builder.Configuration;<br>var services = builder.Services;<br>var openApiKey = configuration[&quot;OpenAI:ApiKey&quot;]!;<br><br>services.AddOpenAIChatClient(&quot;gpt-4o&quot;, openApiKey);<br>services.AddSingleton&lt;StoryWriterAgent&gt;();<br><br>// Setup the Elsa Workflows Engine.<br>services.AddElsa(elsa =&gt;<br>{<br>    elsa.UseWorkflowManagement(management =&gt;<br>    {<br>        management.UseEntityFrameworkCore(ef =&gt; ef.UseSqlite());<br>        management.UseCache();<br>    });<br>    elsa.UseWorkflowRuntime(runtime =&gt;<br>    {<br>        runtime.UseDistributedRuntime();<br>        runtime.UseEntityFrameworkCore(ef =&gt; ef.UseSqlite());<br>        runtime.UseCache();<br>    });<br>    elsa.UseWorkflowsApi();<br>    elsa.UseIdentity(identity =&gt;<br>    {<br>        identity.UseAdminUserProvider();<br>        identity.TokenOptions = options =&gt; options.SigningKey = &quot;secret-signing-key-4d28f39b-7761-42ea-8985-a38faaba4b2d&quot;;<br>    });<br>    elsa.UseDefaultAuthentication();<br>    elsa.AddActivitiesFrom&lt;Program&gt;();<br>});<br><br>// Configure CORS to allow Elsa Studio to invoke the APIs.<br>services.AddCors(cors =&gt; <br>    cors.AddDefaultPolicy(policy =&gt; <br>        policy.AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod().WithExposedHeaders(&quot;*&quot;)));<br><br>// Configure the HTTP request pipeline.<br>var app = builder.Build();<br><br>app.UseCors();<br>app.UseAuthentication();<br>app.UseAuthorization();<br>app.UseWorkflowsApi();<br><br>app.MapGet(&quot;/&quot;, () =&gt; &quot;Hello World!&quot;);<br>app.MapPost(&quot;/write-story&quot;, async (StoryWriterAgent agent, CancellationToken cancellationToken) =&gt; <br>    await agent.WriteStoryAsync(&quot;A haunted lighthouse&quot;, &quot;thriller&quot;, cancellationToken));<br><br>app.Run();</pre><p>With this in place, our ASP.NET Core application now doubles as a workflow server.</p><h4>Wrapping the agent as an Elsa activity</h4><p>One option is to wrap the agent in a custom activity:</p><pre>public class WriteStory : CodeActivity&lt;string&gt;<br>{<br>    public Input&lt;string&gt; Topic { get; set; } = null!;<br>    public Input&lt;string&gt; Genre { get; set; } = null!;<br>    <br>    protected override async ValueTask ExecuteAsync(ActivityExecutionContext context)<br>    {<br>        var agent = context.GetRequiredService&lt;StoryWriterAgent&gt;();<br>        var genre = Genre.Get(context);<br>        var topic = Topic.Get(context);<br>        <br>        var story = await agent.WriteStoryAsync(topic, genre, context.CancellationToken);<br>        context.SetResult(story);<br>    }<br>}</pre><p>This works, but the activity itself doesn’t add any new capability. It’s mostly boilerplate.</p><p>Which raises an interesting question: what if we didn’t have to write activity code at all?</p><h3>Milestone 4: Using agents as activities</h3><p>Elsa 3.6 introduces <strong>Activity Hosts</strong>, which allow public methods on regular C# classes to be exposed directly as workflow activities.</p><p>Registering the agent is as simple as:</p><pre>elsa.AddActivityHost&lt;StoryWriterAgent&gt;();</pre><p>That’s it. The WriteStoryAsync method now shows up as an activity in Elsa Studio, with its method arguments automatically mapped to activity inputs.</p><p>This drastically lowers the friction between application code and workflow orchestration.</p><p>To see this in action, start an instance of Elsa Studio and connect it to our running application (I am running it on port 5001) using the following commands:</p><pre>docker pull elsaworkflows/elsa-studio-blazor-wasm-app:3.6.0<br>docker run -t -i -e ASPNETCORE_ENVIRONMENT=&#39;Development&#39; -e HTTP_PORTS=8080 -e BACKEND__URL=https://localhost:5001/elsa/api -p 14000:8080 elsaworkflows/elsa-studio-blazor-wasm-app:3.6.0</pre><blockquote>Elsa Studio is a separate application that acts as the UI for Elsa Server and communicates with it via HTTP requests.</blockquote><p>When the container has started, navigate to <a href="http://localhost:14000:">http://localhost:14000</a> and login with the default username and password:</p><pre>username: admin<br>password: password</pre><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*xcQzlFyeui_ECHiUMVa1ug.png" /><figcaption>Elsa Studio 3.6 Login</figcaption></figure><p>Once logged in, create a new workflow and add the <em>Write Story</em> activity to the canvas:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*WvMheAeFhDlvx6dGlKbIFw.png" /><figcaption>Our StoryWriterAgent’s <strong>WriteStory</strong> method made available as an activity.</figcaption></figure><p>Notice that the activity’s name is derived from the public methods exposed by the StoryWriterAgent class. In our example, we have one method called WriteStoryAsync. Notice also that the two arguments topic and genre are made available as activity input fields, instantly making our class configurable through Elsa Workflows.</p><h3>Milestone 5: Moving orchestration into Elsa</h3><p>If Elsa is already orchestrating your process, the next logical step is to let it orchestrate the agents themselves.</p><p>Instead of a single StoryWriterAgent, we can split responsibilities into focused agents:</p><pre>public class WriterAgent { … }<br>public class EditorAgent { … }</pre><p>These agents can be registered the same way and surfaced as workflow activities. Properties and method arguments become inputs automatically.</p><p>This is the architectural pivot point: instead of agents coordinating each other, Elsa becomes the primary orchestrator, and agents focus on doing one thing well.</p><h3>Agent Framework workflows vs Elsa workflows</h3><p>So when should you use which?</p><p>Agent Framework workflows are optimized for AI-centric orchestration. They shine when control flow is driven by model decisions, reasoning, and collaboration between agents. If AI behavior <em>is</em> the system, this is a great fit.</p><p>Elsa workflows are optimized for application orchestration. They excel at long-running, event-driven, and stateful processes with explicit control flow, persistence, and operational visibility. In this model, AI agents are participants in a larger, deterministic process.</p><p>In short: use Agent Framework when you are building an AI system, and Elsa when you are building a system that <em>uses</em> AI. They solve orchestration at different layers and complement each other naturally.</p><h3>Closing thoughts</h3><p>Microsoft Agent Framework gives .NET developers a powerful way to build cognition into their applications, with agents that have clear responsibilities, reasoning, and the ability to collaborate. Elsa provides a robust and approachable workflow engine that excels at modeling real-world processes and exposing them through a visual designer.</p><p>Agent Framework shines when orchestration itself is the intelligence.<br>Elsa shines when intelligence is one part of a larger, long-running process.</p><p>What this walkthrough shows is that you don’t have to choose. Agent Framework lets you design rich agentic capabilities, and Elsa gives you a place to put them. Together, they enable systems that are both intelligent and predictable, which becomes especially compelling for scenarios like content pipelines, approvals, and human-in-the-loop processes where intelligence and process need to evolve together.</p><p>Source code of the sample project can be found <a href="https://github.com/elsa-workflows/elsa-samples/tree/release/3.6.0/src/aspnet/Elsa.Samples.AspNet.CodeFirstAgents">here</a>.</p>
