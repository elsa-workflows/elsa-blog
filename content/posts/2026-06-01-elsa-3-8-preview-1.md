# Elsa 3.8 Preview 1

Elsa 3.8.0 preview 1 is not a small release.

There are workflow runtime changes, security hardening items, designer work, dependency updates, and a fair amount of foundation work that will matter more over time than it does on the first read-through of the release notes.

But the part I want to call out here is more focused: Elsa is getting better at being operated.

That sounds less exciting than a new activity or a new designer surface, but it is the kind of work that matters once workflows move beyond local demos. Long-running processes need visibility. Production systems need safer ways to handle credentials. Operators need a way to inspect what happened, and occasionally to correct a running workflow without pretending the only answer is "restart everything and hope".

This preview adds a few pieces in that direction across `elsa-core` and `elsa-studio`:

- Structured Logging UI
- Console Log UI
- OpenTelemetry UI
- Secrets module and UI
- Alterations UI

The release notes are here:

- [elsa-core 3.8.0-preview1](https://github.com/elsa-workflows/elsa-core/releases/tag/3.8.0-preview1)
- [elsa-studio 3.8.0-preview1](https://github.com/elsa-workflows/elsa-studio/releases/tag/3.8.0-preview1)

## Structured logs are now a first-class diagnostics surface

Structured logs are for semantic `ILogger` events: level, category, message template, rendered message, properties, scopes, exception details, tenant, workflow context, correlation IDs, trace IDs, span IDs, and source metadata.

That is a very different thing from tailing stdout.

In Core, the new `Elsa.Diagnostics.StructuredLogs` module captures `ILogger` events, redacts sensitive values, keeps a bounded recent-log buffer, exposes REST endpoints for recent logs, sources and storage diagnostics, and streams live events to Studio over SignalR.

The API surface is intentionally operator-shaped:

```csharp
services.AddElsa(elsa =>
{
    elsa.UseStructuredLogs(options =>
    {
        options.RecentLogCapacity = 5_000;
        options.MaxRecentLogQuerySize = 1_000;
        options.SourceHeartbeatTimeout = TimeSpan.FromSeconds(30);
    });
});

app.UseStructuredLogs();
```

The default store is in memory, which is fine for local development and single-process diagnostics. The module is storage-provider neutral though, and this preview also includes SQLite structured log persistence through `Elsa.Diagnostics.StructuredLogs.Persistence.Sqlite`.

Studio adds the paired viewer at:

```text
/diagnostics/structured-logs
```

The page loads a recent backfill, subscribes to live events, and supports the filters you normally need when a workflow incident is not conveniently happening in a clean local process. You can filter by level, category, message, tenant, workflow instance, source, trace ID and span ID. Workflow instance screens can deep-link into the structured logs page with the workflow filter already applied.

The small but important detail is source awareness. In a containerized or clustered environment, "the server" is not always one process. The structured logs module tracks source metadata such as pod, namespace, container, node, machine name and process ID when available. Studio can then show where the event came from instead of flattening everything into one anonymous stream.

There is also storage diagnostics. If a durable log store is under pressure and starts dropping writes, Studio can surface that rather than silently giving the operator a false sense of completeness.

## Console logs cover the raw stream

Console logs solve a different problem.

Sometimes you do not want semantic logging. You want the thing the process wrote to stdout or stderr. Maybe it is an external tool. Maybe it is a quick `Console.WriteLine` while debugging a workflow. Maybe it is output that includes ANSI colors and already has the shape the developer expects.

Core now has `Elsa.Diagnostics.ConsoleLogs`, built on top of `ConsoleLogStreaming.Core`, with Elsa-specific workflow metadata layered in. It captures raw stdout and stderr from the current backend process, keeps recent bounded history, streams live updates over SignalR, exposes sources over REST, and reports dropped-line summaries when buffers overflow.

Configuration looks like this:

```csharp
services.AddElsa(elsa =>
{
    elsa.UseConsoleLogs(options =>
    {
        options.RecentCapacity = 5_000;
        options.SubscriberCapacity = 1_000;
        options.MaxRecentQuerySize = 1_000;
        options.MaxLineLength = 16_384;
        options.PreserveAnsi = true;
    });
});

app.UseConsoleLogs();
```

The Studio module contributes a diagnostics page at:

```text
/diagnostics/console
```

and also adds workflow instance console-log widgets, so you can inspect console output in the context of the workflow instance you are looking at.

This separation between structured logs and console logs is deliberate. Structured logs are queryable records with fields. Console logs are raw lines. They have different failure modes and different uses. Mixing them into one UI tends to make both worse.

In practice, I expect console logs to be useful during development, support sessions, and operational debugging where "what did the process just print?" is the fastest question to answer. Structured logs are the better fit when you need filtering, correlation, and repeatable diagnostics.

## OpenTelemetry gets a Studio diagnostics view

The OpenTelemetry work in this preview is also split cleanly between Core and Studio.

Core adds `Elsa.Diagnostics.OpenTelemetry`, which owns the collector/read side:

- OTLP HTTP/protobuf ingestion
- gRPC collector registration
- a bounded in-memory diagnostics store
- resources, traces, metrics and logs query APIs
- storage diagnostics
- collector configuration metadata
- live updates over SignalR
- read permissions

The default OTLP HTTP endpoint is:

```text
/elsa/otlp/v1
```

with the usual signal-specific paths below it:

```text
POST /elsa/otlp/v1/traces
POST /elsa/otlp/v1/metrics
POST /elsa/otlp/v1/logs
```

The Studio-facing read APIs live under diagnostics routes such as:

```text
/diagnostics/opentelemetry/traces/search
/diagnostics/opentelemetry/metrics/search
/diagnostics/opentelemetry/logs/search
/diagnostics/opentelemetry/storage
/diagnostics/opentelemetry/collector-configuration
```

Studio then adds the UI at:

```text
/diagnostics/opentelemetry
```

The page is a viewer for normalized Core DTOs. That point matters. Studio does not parse OTLP protobuf payloads and does not try to become a telemetry collector. Core owns ingestion, redaction, storage, permissions and contracts. Studio reads the normalized model and renders resources, traces, metrics, OTLP logs, storage pressure and collector setup.

There is a trace waterfall component, resource picker, metric series table, span details drawer, live update controls, export action, filters, and storage overflow state. The UI also keeps OpenTelemetry separate from structured logs and console logs, while still leaving room for correlation through public route/query contracts using trace, span, resource and time values.

This is not intended to replace your production observability platform.

The default store is bounded in memory. It is there so an Elsa host can expose recent telemetry to Studio without requiring a full observability stack for every environment. If you already run OpenTelemetry into a vendor or collector pipeline, this does not compete with that. It gives Elsa Studio a direct operational view over recent workflow-relevant telemetry.

## Secrets move into the product model

Secrets are one of those features that start out as "we can just use configuration" and eventually turn into a mess of copied values, hidden environment variables, and workflow inputs that should never have been treated like ordinary text.

This preview adds a first-party `Elsa.Secrets` module in Core and a matching Studio UI.

The Core module provides:

- named secret management
- secret type descriptors
- secret store descriptors
- create, list, get, update, rotate, revoke, delete and test endpoints
- a picker endpoint for UI integration
- encrypted and configuration-backed stores
- file and in-memory repositories
- EF Core persistence packages
- a `Secret` expression type
- JavaScript `getSecret(name)` support through `Elsa.Secrets.JavaScript`

The model is intentionally more than a name/value bag. A secret has a name, display name, description, type, store, optional scope, tags, status, current version, timestamps and expiration. Rotation creates a new value version. Revoke and delete are explicit operations. The test endpoint resolves the secret and reports whether that succeeded without turning the UI into a place where the value has to be displayed.

Studio adds the Secrets page under settings:

```text
/security/secrets
```

From there you can search, create, open details, rotate, test, revoke and delete secrets. More importantly, Studio also adds a secret picker UI hint. Inputs marked as secret-capable can render a picker instead of a plain text box, and wrapped inputs can store a `Secret` expression reference rather than the secret value itself.

That changes the workflow authoring story in a small but meaningful way.

Instead of pasting an API key into an activity input, the workflow can hold a reference to a named secret. At runtime, the resolver gets the value from the configured store. JavaScript expressions can call `getSecret("name")` when that feature is enabled.

This is still a preview, so I would treat it as the product foundation rather than the final word on enterprise secret management. The important part is that Elsa now has a native contract for secret references, secret-aware input UI, and runtime resolution. That gives us a proper place to integrate with more stores and policies over time.

## Alterations get a Studio workflow

Alterations are about changing running workflow instances deliberately.

That is a dangerous sentence if the system does not give you a careful model for it. But in real workflow systems, operators eventually need this kind of capability. A workflow may be stuck on an activity that should be cancelled. A variable may need correction. An instance may need to move to a newer definition version. An activity may need to be scheduled because the outside world already moved on.

Core already has the alteration concepts in this preview:

- alteration plans
- alteration jobs
- dry run, run, submit and get endpoints
- retry workflow support
- plan and job stores
- plan scheduling and background dispatch
- built-in alteration types for cancel, cancel activity, schedule activity, modify variable and migrate

Studio adds the missing operator experience.

There is an Alterations module with pages for alteration plans, workflow instance selection, plan details, and an instance edit surface. The designer side has an alteration catalog, staging service, side panel, configuration dialog and submit dialog. The catalog exposes the built-in operations as understandable actions: cancel workflow, cancel activity, schedule activity, modify variable, and migrate to a newer version.

The workflow here is intentionally staged. You select or inspect a workflow instance, add one or more intended alterations, configure their fields, review them, and submit a plan. Plan details then show the resulting jobs, statuses and log entries.

That is the right shape for this feature. Runtime correction should not feel like clicking a random button on a live process. It should feel like preparing an operational change and then applying it with enough context to understand what happened afterwards.

## The theme of this preview

The obvious headline is "new UI modules", but I think the more useful reading is this:

Elsa 3.8 preview 1 makes more runtime state visible and more operational actions explicit.

Structured logs give you semantic application events.
Console logs give you raw stdout and stderr.
OpenTelemetry gives you resources, traces, metrics and OTLP logs.
Secrets give workflows a native way to refer to sensitive values without storing them as ordinary input text.
Alterations give operators a controlled path for correcting running workflow instances.

None of these remove the need to design workflows carefully. They also do not remove the need for production-grade observability, secrets policy, backups, permissions, and review processes.

But they do move Elsa in the right direction for real systems.

Workflow engines do not become useful in production merely because they can execute a process. They become useful when teams can see what is happening, understand why something failed, protect the values that should not leak, and make careful corrections when reality disagrees with the ideal path.

That is the interesting part of this preview.
