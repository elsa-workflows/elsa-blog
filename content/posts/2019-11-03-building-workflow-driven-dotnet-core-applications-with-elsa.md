<section><div><div><h3>Building Workflow Driven .NET Core Applications with Elsa</h3><figure><img data-image-id="1*8p8UkoB3Bj0A5e1IAtUZsw.png" data-width="3840" data-height="559" data-is-featured="true" src="https://cdn-images-1.medium.com/max/800/1*8p8UkoB3Bj0A5e1IAtUZsw.png"></figure><blockquote>This article is about Elsa 1. If you want to learn how to build workflow driven .NET applications with Elsa 2, please checkout <a href="https://sipkeschoorstra.medium.com/building-workflow-driven-net-applications-with-elsa-2-part-1-44e08a9ba94b" rel="noopener" target="_blank">this series</a> instead.</blockquote><p>For many years, I’ve been developing .NET applications using <a href="https://orchardproject.net/" rel="noopener" target="_blank">Orchard</a>, an extensible, modular, open source application framework and CMS built with Classic ASP.NET MVC back in the days, and now <a href="https://orchardcore.readthedocs.io/en/dev/" rel="noopener" target="_blank">redeveloped for ASP.NET Core</a>.</p><p>One if its most awesome features is <strong>Workflows</strong>. I’ve used Orchard 1 workflows to implement user registration, authentication and content approval workflows. Being able to create, configure and modify complex, long-running business processes is a powerful feat in many an app, and the Orchard Workflow module makes this sort of thing child’s play.</p><p>I often found myself wishing that I could use it in other ASP.NET Core applications as well. However, these applications were not Orchard Core powered applications, nor did the simple desire to use workflows warrant taking a dependency on Orchard Core Framework.</p><p>Ultimately, I decided it was about time to take Orchard Core’s workflow engine and philosophies, and turn it into a reusable, .NET Standard library, enabling *any* .NET application to enjoy the power of workflow.</p><p>Meet <a href="https://elsa-workflows.github.io/elsa-core/" rel="noopener" target="_blank">Elsa</a>, an open source suite of .NET Standard libraries and tools that enable developers to implement long-running workflows. The core philosophy of Elsa is to be able to connect small executable units with one another, allowing you to orchestrate real business processes such as document approval, customer on-boarding and order fulfillment processes.</p><p>As should be clear by now, Elsa is inspired on Orchard Core Workflows. But I also took inspiration from a number of other existing workflow solutions, such as <a href="https://docs.microsoft.com/en-us/dotnet/framework/windows-workflow-foundation/overview" rel="noopener" target="_blank">Windows Workflow Foundation</a> and <a href="https://azure.microsoft.com/services/logic-apps/" rel="noopener" target="_blank">Azure Logic Apps</a>, and I continue to be inspired by libraries such as <a href="https://github.com/danielgerlag/workflow-core" rel="noopener" target="_blank">Workflow Core</a> and <a href="https://workflowengine.io/" rel="noopener" target="_blank">Workflow Engine</a>.</p><p>Elsa can be used to implement a workflow engine within your own application’s process, or as an external workflow service with which your applications interact with. Furthermore, workflows can be <a href="https://elsa-workflows.github.io/elsa-core/docs/guides-hello-world-http" rel="noopener" target="_blank">implemented as strongly typed classes</a>, as JSON/YAML files, or visually using a <a href="https://elsa-workflows.github.io/elsa-core/docs/guides-dashboard#create-a-workflow" rel="noopener" target="_blank">HTML5 web-based designer</a>.</p><p>To illustrate its potential, Let’s walk through a simple scenario: <em>User signup</em>.</p><blockquote>The remainder of this post is a walkthrough of setting up a Blazor application, integrating it with workflow services and then creating and running a workflow. If it’s too long to read, you can skip ahead and download <a href="https://github.com/elsa-workflows/elsa-samples/tree/master/src/Elsa.Samples.UserRegistration" rel="noopener" target="_blank">the final solution</a> and play around with the workflow editor. Make sure to import the accompanying <a href="https://github.com/elsa-workflows/elsa-samples/blob/master/src/Elsa.Samples.UserRegistration/user-registration-workflow.json" rel="noopener" target="_blank">workflow json file</a> that you can import from the workflow editor view.</blockquote><h3>Scenario: User Signup</h3><p>In this scenario, we have a web application where users can <strong>register</strong> an account. Let’s consider what we’re trying to do:</p><ol><li>We want to trigger a workflow when the user submits the registration form.</li><li>The workflow should create a User record.</li><li>The workflow should send the user an activation link via email.</li><li>The workflow should wait for the user to click the activation link.</li><li>When the user clicks the activation link, we want the workflow to resume and actually activate the user.</li><li>When the user is activated, we want to send a confirmation email.</li><li>If the user does not click the activation link within 5 minutes, we want to delete the user.</li><li>When the user is deleted, we will send an email to the user saying that we’re sorry to see them go.</li><li>When 2.5 minutes have passed since sending the activation link, we want to send a reminder via email encouraging the user to activate their account by stressing them that time is running out.</li></ol><p>Let’s start with #1.</p><h4>Trigger workflow when user submits registration form</h4><p>There are two activities that we could choose for this purpose:</p><ol><li>Receive HTTP Request</li><li>Signaled</li></ol><p>The <strong>Receive HTTP Request</strong> activity makes sense, because it allows us to define a URL to which the activity will respond, and we can configure the form element in our view to simply post to that activity, and that would be that.</p><p>However, I will show you another way that demonstrates how we can trigger a workflow using <code>IWorkflowInvoker</code> and the <strong>Signaled </strong>activity.</p><p>The <strong>Signaled</strong> activity is a blocking activity that can be used as a workflow trigger. This means the workflow will start when this signal is triggered.</p><p>Let’s go with this. We also pass as an input value the <code>registrationModel</code>, which will contain the submitted form values thanks to Blazor’s binding mechanism.</p><p>Keep this workflow input in mind, because we will use it next when we add the second activity to the workflow.</p><h4>Create User</h4><p>Go back to the workflow editor and this time add the <strong>Create User</strong> activity we created earlier.</p><figure><img data-image-id="1*RWsrq24C6a63WXFA3H26wQ.png" data-width="1705" data-height="596" src="https://cdn-images-1.medium.com/max/800/1*RWsrq24C6a63WXFA3H26wQ.png"></figure><p>When added to the canvas, double-click this activity to open the activity editor again. This time, we will enter all the values necessary to create a user. And that’s where things get really interesting.</p><p>Obviously, we don’t want to hard-code a bunch of values here. On the contrary, we want to use the values that were submitted through the form that initiates this workflow. So how do we get those values?</p><p>Enter Workflow Expressions.</p><p>As stated earlier, Elsa supports workflow expressions, and several syntaxes at that.</p><p>Let’s see how that works.</p><p>Since we trigger this workflow while providing Input variables, we can use the following Liquid expression to access such variables:</p><pre>{{ Input.SomeVariableName }}</pre><p>Since we invoke the workflow with an input variable called <code>RegistrationModel</code>, we will be able to access it using the following expression:</p><pre>{{ Input.RegistrationModel }}</pre><p>Sweet! However, there’s a small caveat: Fluid, the library providing us with this Liquid goodness, implements a white-listing strategy, which means that nothing is accessible by default. To make things accessible, such as our <code>RegistrationModel</code>, we need to tell Fluid so.</p><p>To configure Liquid with access to certain types, we need to whitelist these types. We do this by implementing a notification handler for the <code>EvaluatingLiquidExpression</code> message, which Elsa publishes when it’s about to evaluate a Liquid expression.</p><p>That’s where the <code>LiquidConfigurationHandler</code> class comes into play we defined earlier.</p><p>Since the <code>RegistrationModel</code> variable is of type <code>RegistrationModel</code>, we can now access all of its properties from Liquid expressions.</p><p>Enter the following values for the Create User activity:</p><ul><li>User Name: <code>{{ Input.RegistrationModel.Name }}</code></li><li>Email: <code>{{ Input.RegistrationModel.Email }}</code></li><li>Password: <code>{{ Input.RegistrationModel.Password }}</code></li><li>Name: <code>CreateUser</code></li></ul><p>Make sure to set the syntax for each field (where applicable) to <strong>Liquid</strong>.</p><p>Notice that we also filled out the <strong>Name</strong> field. Each activity has an optional <strong>Name</strong> field that we can configure with a name of our own choosing. Doing so allows us to reference this activity by name from workflow expressions.</p><p>This ability is key, because it allows us to access the user object that will be created by the <strong>Create User</strong> activity.</p><p>We’ll see how to do this next.</p><h4>Send Activation Link via Email</h4><p>Now that the user is created, we need to send them an email with an activation link. Once the email is sent, we want the workflow to pause and wait for the user to click the activation link, after which the workflow should resume and actually activate the user. How to make it work?</p><p>As it turns out, it’s quite easy:</p><ol><li>We generate a URL that will trigger the workflow using a signal name of our choosing. Let’s call this signal <code>“Activate”</code>.</li><li>We add another <strong>Signaled</strong> activity to the workflow that will cause the workflow to be suspended until its configured signal name is triggered (which will happen when the generated URL is invoked)</li></ol><p>OK. But how do we generate that URL, you wonder. Well, let’s have a look.</p><p>Add a <strong>Send Email</strong> activity and configure its properties with the following values:</p><ul><li>From: <code>noreply@acme.com</code></li><li>To: <code>{{ Activities.CreateUser.User.Email }}</code></li><li>Subject: <code>Activate your account!</code></li><li>Body:</li></ul><pre>&lt;p&gt;Welcome, {{ Activities.CreateUser.User.Name }}!&lt;/p&gt;<br>&lt;p&gt;Please &lt;a href=”{{ ‘Activate’ | signal_url }}”&gt;activate your account &lt;/a&gt; within the next 5 minutes.&lt;/p&gt;</pre><ul><li>Name: &lt;blank&gt;</li><li>Title: <code>Send Activation Link Email</code></li><li>Description: <code>Send an email containing an activation link.</code></li></ul><p>Again, make sure to select the Liquid syntax for those fields where you entered Liquid expressions.</p><p>Let’s have a closer look at the Body field’s contents, specifically the value of the href attribute of the activation link:</p><p><code>href=”{{ ‘Activate’ | signal_url }}”</code></p><p>What we see here is a Liquid expression where we simply render the <code>“Activate”</code> string and then apply the <code>signal_url</code> Liquid filter. This filter will take the <code>“Activate”</code> string as an input and generate an absolute URL that will trigger the workflow with the specified signal name.</p><blockquote>In case you are wondering how this will work when multiple workflow instances are executing, here’s how: when a workflow starts, what really happens is that a workflow instance is created from the workflow definition (the workflow we are designing). The generated signal URL contains an encrypted query string value carrying the signal name as well as the <strong>workflow instance ID</strong>. That’s how the HTTP middleware handling the incoming URL knows which workflow instance to resume.</blockquote><p>Now that we have sent the email containing an activation link, we need to wait for the user to click on it.</p><p>But we don’t want to wait indefinitely. As per the acceptance criteria, we will allow the user 5 minutes within which they must activate their account before we delete it.</p><p>If you are thinking: “sounds like we need to fork this thing”, you are correct: fork is right.</p><h4>Fork</h4><p>Open the Activity Picker, look for the <strong>Fork</strong> activity and add it to the workflow. Then double-click the activity and enter the following values for each field:</p><ul><li>Branches: <code>Activate, Timeout</code></li><li>Name: &lt;blank&gt;</li><li>Title: &lt;blank&gt;</li><li>Description: <code>Split execution</code></li></ul><p>And click <strong>Save</strong>.</p><p>Notice that the Fork activity has two outcomes: <strong>Activate</strong> and <strong>Timeout</strong>, both of which we defined in the <strong>Branches</strong> field of the activity.</p><p>The Fork activity allows us to specify any number of branches using any name we like, separated by a comma. When the Fork activity executes, it will split workflow execution into all of the specified branches, and all of these branches will be executed.</p><blockquote>I would say “executed in parallel”, but in reality the workflow invoker executes each branch one after the other; however this is an implementation detail, and might change and actually execute the branches in parallel in the future. From a logical point of view, you can consider each branch to execute in parallel.</blockquote><p>Let’s implement the <strong>Activate branch</strong> first. When we’re done with that, we will circle back and implement the <strong>Timeout branch</strong>.</p><h4>Activate Branch: Activation Link Clicked</h4><p>Add a new activity of type <strong>Signaled</strong> and provide it with the following values:</p><ul><li>Signal: <code>Activate</code></li><li>Name: &lt;blank&gt;</li><li>Title: <code>Activation Link Clicked</code></li><li>Description: <code>Wait for user to click the activation link.</code></li></ul><p>Notice that the signal name is the exact same value as the name we used when generating the signal URL.</p><p>With this activity in place, workflow execution will simply halt and be resumed once it receives the <strong>Activate</strong> signal.</p><p>Make sure to connect the Fork activity to this activity (using the Activate outcome).</p><h4>Activate Branch: Activate User</h4><p>When the workflow does receive the <strong>Activate</strong> signal, we want to activate the user. This is where our <strong>Activate User</strong> activity comes into play. Open the activity picker, look for the <strong>Activate User</strong> activity, and add it to the workflow.</p><p>Provide the activity with the following values:</p><ul><li>User ID: <code>{{ Activities.CreateUser.User.Id }}</code></li><li>Name: &lt;blank&gt;</li><li>Title: &lt;blank&gt;</li><li>Description: <code>Activate the created user.</code></li></ul><p>Notice that we again rely on a Liquid expression to access the output of another activity in the workflow (the named <strong>CreateUser</strong> activity in this case).</p><p>When this activity executes, it will simply switch the <code>IsActivate</code> property of the created user to <code>true</code> and store that change.</p><p>Now that the user is activated, let’s send them an email to let them know.</p><h4>Activate Branch: Send Activation Confirmation Email</h4><p>Add another <strong>Send Email</strong> activity and provide the following values:</p><ul><li>From: <code><a href="mailto:noreply@acme.com" rel="nofollow" target="_blank">noreply@acme.com</a></code></li><li>To: <code>{{ Activities.CreateUser.User.Email }}</code></li><li>Subject: <code>Thanks for activating!</code></li><li>Body:</li></ul><pre>&lt;p&gt;Hi {{ Activities.CreateUser.User.Name }},&lt;/p&gt;<br>&lt;p&gt;Thanks for activating your account!&lt;/p&gt;</pre><ul><li>Name: &lt;blank&gt;</li><li>Title: <code>Send Activation Confirmation Email</code></li><li>Description: <code>Send an email confirming that the account has been activated.</code></li></ul><p>Nothing we haven’t seen already.</p><p>We’re almost done with this branch. All that’s left is to display a message in the browser window. Remember: the user will have clicked the activation link, taking them to a new browser tab or window, which will invoke the workflow. By default, the HTTP middleware will simply reply with an <code>HTTP Accepted</code> status code. But all the user will see is a blank page, which will confuse the user (“did it work?”). Instead, we should either redirect the user to another page stating that their account has been successfully activated, or at the very least render a message to that effect. I will keep it simple here, so let’s go with the latter and add the <strong>Write HTTP Response</strong> activity.</p><h4>Activate Branch: Display Activation Confirmation Page</h4><p>Open the activity picker, look for the <strong>Write HTTP Response</strong> activity, and provide the following values:</p><ul><li>Status Code: <code>200</code></li><li>Content:</li></ul><pre>&lt;p&gt;Hi {{ Activities.CreateUser.User.Name }},&lt;/p&gt;<br>&lt;p&gt;Thanks for activating your account!&lt;/p&gt;</pre><ul><li>Content Type: <code>text/html</code></li><li>Response Headers: &lt;blank&gt;</li><li>Name: &lt;blank&gt;</li><li>Title: <code>Display Activation Confirmation Page</code></li><li>Description: <code>Display a page confirming that the account has been activated.</code></li></ul><p>That’s it for the Activate branch! Let’s circle back to the Fork activity, and implement the <strong>Timeout branch</strong>.</p><h4>Timeout Branch: Wait for 2.5 minutes</h4><p>The purpose of the <strong>Timeout branch</strong> is to delete the user account if said user does not activate said account within 5 minutes. We also want to send a reminder after 2.5 minutes that time is running out.</p><p>That’s what we will do first: after 2.5 minutes, send a reminder email.</p><p>Open the activity picker and look for the <strong>Timer Event</strong> activity and provide it with the following values:</p><ul><li>Timeout Expression: <code>00:02:30</code></li><li>Name: &lt;blank&gt;</li><li>Title: <code>Wait for 2.5 minutes</code></li><li>Description: &lt;blank&gt;</li></ul><h4>Timeout Branch: Send Reminder Email</h4><p>Add a <strong>Send Email</strong> activity with the following values:</p><ul><li>From: <code><a href="mailto:noreply@acme.com" rel="nofollow" target="_blank">noreply@acme.com</a></code></li><li>To: <code>{{ Activities.CreateUser.User.Email }}</code></li><li>Subject: <code>Don’t forget to activate your account!</code></li><li>Body:</li></ul><pre>&lt;p&gt;Hi {{ Activities.CreateUser.User.Name }}!&lt;/p&gt;<br>&lt;p&gt;Please don’t forget to &lt;a href=”{{ ‘Activate’ | signal_url }}”&gt;activate your account &lt;/a&gt; within the next 2 minutes.&lt;/p&gt;</pre><ul><li>Name: &lt;blank&gt;</li><li>Title: <code>Send Reminder Email</code></li><li>Description: <code>Send a reminder email that the user’s time is running out!</code></li></ul><p>Nothing we haven’t seen before either. Do notice that we are generating a new activation URL. However, it doesn’t matter whether the user clicks the original one, or the one from the reminder email; both URLS carry the proper signal name (“Activate”) and workflow instance ID.</p><blockquote><strong>Tip:</strong> If for whatever reason you prefer to use the exact same URL in both emails, you could add a <strong>Set Variable</strong> activity, and use the <code>signal_url</code> Liquid filter there. The generated URL will now be stored in a workflow variable, which you can reference from both <strong>Send Email</strong> activities.</blockquote><h4>Timeout Branch: Wait for 2.5 minutes</h4><p>After sending the email, the workflow must wait for another 2.5 minutes before calling it a night. Add another <strong>Timer Event</strong> with the exact same values used for the other <strong>Timer Event</strong>.</p><h4>Timeout Branch: Delete User</h4><p>Time’s up, and we are left with no choice but to delete the user. Add the <strong>Delete User</strong> activity and provide it with the following values:</p><ul><li>User ID: <code>{{ Activities.CreateUser.User.Id }}</code></li><li>Name: &lt;blank&gt;</li><li>Title: &lt;blank&gt;</li><li>Description: <code>Delete the created user.</code></li></ul><h4>Timeout Branch: Send Account Deleted Email</h4><p>After the user has been deleted, we will let them know about this sad news by sending them a final email.</p><p>Add a new <strong>Send Email</strong> activity with the following values:</p><ul><li>From: <code><a href="mailto:noreply@acme.com" rel="nofollow" target="_blank">noreply@acme.com</a></code></li><li>To: <code>{{ Activities.CreateUser.User.Email }}</code></li><li>Subject: <code>Sorry to see you go</code></li><li>Body:</li></ul><pre>&lt;p&gt;Hi {{ Activities.CreateUser.User.Name }},&lt;/p&gt;<br>&lt;p&gt;We’re sorry to see you go. We look forward to having you again!&lt;/p&gt;</pre><ul><li>Name: &lt;blank&gt;</li><li>Title: <code>Send Account Deleted Email</code></li><li>Description: <code>Send the user an email confirming that their account has been deleted.</code></li></ul><h4>Finish</h4><p>Way to go! Since you’ve made it this far, let’s go the extra mile and finish up the workflow. By default, a workflow will automatically finish when there are no more blocking activities to resume. But in our workflow, there are two branches of execution: <strong>Activate</strong> and <strong>Timeout</strong>. If the user activates their account within 5 minutes, their account will be activated and that’s fine.</p><p>However, the workflow will still have one blocking activity: one of the two <strong>Timer Events</strong>. And they <em>will </em>eventually trigger! This is bad, as it will cause the user account to be deleted.</p><p>There are a number of ways we can mitigate this:</p><ol><li>Add a <strong>Join </strong>activity that merges the two branches back into a single path of execution.</li><li>Add two <strong>IfElse</strong> activities in the <strong>Timout branch</strong> to check if the user has already been activated. When true, don’t send reminders and don’t delete the user.</li><li>Add a <strong>Finish</strong> activity that removes any and all blocking activities.</li></ol><p>We will go with option #3 because it’s the simplest of all and specifically designed to handle this sort of scenarios. So, go ahead and add the <strong>Finish</strong> activity to the workflow, and make sure to connect both branches of execution to this one.</p><figure><img data-image-id="1*3pDPu_svFi7pLSlGcjcBkQ.png" data-width="1953" data-height="1021" src="https://cdn-images-1.medium.com/max/800/1*3pDPu_svFi7pLSlGcjcBkQ.png"></figure><p>That’s it! Make sure to <strong>Publish</strong> the workflow, and we will take it for a spin!</p></div></div></section><section><div><div><h3>Taking it for a spin</h3><p>Launch the application and make sure you have an SMTP host running (I’m using <strong>Smtp4Dev</strong>).</p><p>We’ll take two spins: first we test the <strong>Timeout branch</strong>, secondly we will test the <strong>Activate branch</strong>.</p><h4>Timeout Branch</h4><p>We start by filling out the registration form:</p><figure><img data-image-id="1*zvfAp6pbhkpmCZWlhBwNKA.png" data-width="1000" data-height="873" src="https://cdn-images-1.medium.com/max/800/1*zvfAp6pbhkpmCZWlhBwNKA.png"></figure><p>When you press <strong>Submit</strong>, you should see a confirmation message stating that you should check your email.
