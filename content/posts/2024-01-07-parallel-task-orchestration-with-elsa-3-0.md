In C#, this can be achieved neatly and efficiently. Here’s how you might approach it:</p><pre>using System.Diagnostics;<br><br>// Setup an HttpClient.<br>var httpClient = new HttpClient();<br><br>// Create a list of tasks.<br>var httpRequestTask1 = httpClient.GetAsync(&quot;https://reqres.in/api/users/1?delay=3&quot;);<br>var httpRequestTask2 = httpClient.GetAsync(&quot;https://reqres.in/api/users/2?delay=3&quot;);<br>var httpRequestTask3 = httpClient.GetAsync(&quot;https://reqres.in/api/users/3?delay=3&quot;);<br><br>// Begin timing.<br>var stopwatch = new Stopwatch();<br>stopwatch.Start();<br><br>// Execute all requests in parallel.<br>var responses = await Task.WhenAll(httpRequestTask1, httpRequestTask2, httpRequestTask3);<br><br>// Print the responses.<br>foreach (var response in responses) <br>    Console.WriteLine($&quot;{response.StatusCode} response from {response.RequestMessage!.RequestUri}:&quot;);<br><br>// Stop timing.<br>stopwatch.Stop();<br><br>// Print the elapsed time.<br>Console.WriteLine($&quot;Finished in {stopwatch.Elapsed.TotalSeconds} seconds&quot;);<br></pre><p>The beauty of this program lies in its parallel execution. In this example, a single HTTP request takes 3 seconds. Running them sequentially would take 9 seconds for the program to complete. However, executing them concurrently, as shown, cuts the total time to just 3 seconds.</p><h3>The Elsa Way</h3><p>Transitioning from C# to Elsa workflows opens up new possibilities in handling parallel tasks, simplifying parallel task orchestration visually. Let’s explore how we can replicate the execution of 3 HTTP requests in parallel, as we did in C#, but this time within an Elsa workflow.</p><h4>Visual Representation</h4><p>The <a href="https://raw.githubusercontent.com/elsa-workflows/elsa-documentation/main/v3/public/guides/parallel-activity-execution/parallel-http-requests.json">following workflow</a> demonstrates the execution of 3 HTTP requests in parallel:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*F4GKi1gPTYRIw7Z5A7QfJQ.png" /><figcaption>The workflow version of executing 3 HTTP requests in parallel</figcaption></figure><h4>Sequential Invocation, Concurrent Execution</h4><p>Although it may appear that Elsa executes these activities in parallel, the engine technically invokes them <em>sequentially</em>. The real magic of concurrency in Elsa lies in how the activities are processed.</p><h4>Achieving True Concurrency</h4><p>True concurrency in Elsa is achieved by allowing activities to perform work in the background. Here’s how it’s done:</p><p><strong>1. Asynchronous Configuration of Activities:</strong> In the case of an HTTP Request activity, you can set it to run asynchronously. This is visually represented in Elsa by toggling the “Run Asynchronously” option:</p><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*VN3F-hbkrm_cpGrmnauAcA.png" /><figcaption>Task activities such as HTTP Request can be configured to perform its work from the background</figcaption></figure><p><strong>2. Sequential Scheduling, Concurrent Execution:</strong> When an activity like an HTTP Request is encountered in the workflow, it’s scheduled for execution as usual. However, if “Run Asynchronously” is checked, the engine does something clever:</p><ul><li>It creates a background job and associates it with a bookmark.</li><li>This process is quick, allowing the engine to move on and schedule the next activity, which might also be set to run asynchronously.</li></ul><p><strong>2. Concurrent Jobs:</strong> In our scenario, this means three jobs are scheduled to run concurrently.</p><p><strong>3. Completion and Bookmark Resumption:</strong> Once a job completes, it resumes the workflow using its bookmark, effectively synchronising the parallel processes.</p><h4>The Impact of Asynchronous Execution</h4><p>Without the “Run Asynchronously” setting, the entire workflow would take roughly 9 seconds to complete, assuming each HTTP request takes 3 seconds. However, with asynchronous execution enabled, the workflow can complete in about 3 seconds. This demonstrates the power of Elsa in handling parallel tasks efficiently, mimicking the concurrency we achieved in C# but in a workflow-based environment.</p><h3>Seeing it in Action</h3><p>One of the best ways to appreciate the power of parallel task execution in Elsa is to witness it firsthand. Follow these steps to set up an environment where you can see the significant difference that asynchronous execution makes.</p><ol><li><strong>Get the Workflow File</strong>: Download <a href="https://raw.githubusercontent.com/elsa-workflows/elsa-documentation/main/v3/public/guides/parallel-activity-execution/parallel-http-requests.json">this workflow file</a>.</li><li><strong>Set Up Elsa Server + Studio</strong>: Fetch the latest version of Elsa Server + Studio Docker container and run it with the following command:</li></ol><pre>docker pull elsaworkflows/elsa-server-and-studio-v3:latest<br>docker run -t -i -e ASPNETCORE_ENVIRONMENT=&#39;Development&#39; -e HTTP_PORTS=8080 -e HTTP__BASEURL=http://localhost:13000 -p 13000:8080 elsaworkflows/elsa-server-and-studio-v3:latest</pre><p>3.<strong> Access Elsa Studio</strong>: Navigate to <a href="http://localhost:13000/">http://localhost:13000/</a></p><p>4.<strong> Log In</strong>: Use the default credentials:</p><pre>username: admin<br>password: password</pre><p>5.<strong> Import the Workflow</strong>:</p><ul><li>Select <em>Workflows</em> -&gt; <em>Definitions</em> from the menu</li><li>Import the file you downloaded in step 1.</li></ul><p>6.<strong> Open and Execute the Workflow</strong>:</p><ul><li>Select the imported workflow to open it in the workflow designer.</li><li>Click the green play button to execute the workflow. Observe that it completes in approximately 3 seconds.</li></ul><p>7. <strong>Modify and Re-Test</strong>:</p><ul><li>Update each HTTP Request activity by selecting them and unchecking the “Run Asynchronously” setting.</li><li>Re-execute the workflow as in step 6. You’ll notice a stark difference: the execution time jumps to around 9 seconds.</li></ul><h3>Other Jobs</h3><p><em>Expanding the Scope of Concurrent Execution in Elsa</em></p><p>Elsa workflows aren’t limited to just executing HTTP requests in parallel. The frameworks versatility extends to a wide range of activities, and it even supports <em>custom activities with background execution</em>. This capability significantly broadens the potential applications of Elsa in various scenarios.</p><p>Writing your own activity that supports background execution is straightforward in Elsa. This can be done by specifying the kind of activity as either Task or Job.</p><p>For example:</p><pre>[Activity(&quot;MyNameSpace&quot;, &quot;MyCategory&quot;, &quot;My description&quot;, Kind = ActivityKind.Task)]<br>public class MyCustomTaskActivity : CodeActivity<br>{<br>}<br><br>[Activity(&quot;MyNameSpace&quot;, &quot;MyCategory&quot;, &quot;My description&quot;, Kind = ActivityKind.Job)]<br>public class MyCustomJobActivity : CodeActivity<br>{<br>}</pre><p>The key distinction lies in their execution control:</p><ul><li><strong>Task Kind</strong>: Offers the user control over whether to run the activity in the background.</li><li><strong>Job Kind</strong>: Automatically runs in the background, without user intervention.</li></ul><p>By harnessing the power of Tasks and Jobs, workflows can be tailored to a wide variety of parallel processing needs, from simple HTTP requests to complex, custom-defined operations.</p><h3>Horizontal Scalability</h3><p>A key difference between using Elsa concurrent activity execution and the C# example earlier, is the approach to task dispatching. Elsa leverages a queue-based system, which can range from simple in-memory queues for less complex tasks to robust service bus queues like Azure Service Bus and RabbitMQ for larger-scale applications. This method contrasts sharply with the direct execution approach seen in C#, where all three tasks are executing in-memory in the same application instance.</p><p>The key advantage of this system is its ability to scale out horizontally. As workload demands increase, more resources or nodes can be added, significantly enhancing the system’s capacity to handle a higher volume of workflows efficiently. This approach not only ensures a more robust and reliable execution of tasks but also caters to scalable solutions, making Elsa a compelling framework for optimising workflow throughput in diverse operational scales and complexities.</p><h3>Summary</h3><p>Elsa 3’s parallel task execution capabilities significantly reduces overall execution time when there is a need for running &amp; orchestrating multiple jobs.</p><p><strong>Key Highlights</strong>:</p><ol><li><strong>Parallel Execution in C#</strong>: We began by illustrating how multiple HTTP requests can be executed concurrently in C#, a method that significantly cuts down on execution time compared to sequential processing.</li><li><strong>The Elsa Workflow</strong>: Transitioning from C# to Elsa, we demonstrated how similar parallel execution can be achieved in workflow environments. By leveraging the “Run Asynchronously” setting in Elsa, tasks that would normally take longer to complete sequentially are executed in a fraction of the time.</li><li><strong>Beyond HTTP Requests</strong>: Elsa’s capabilities are not limited to HTTP requests. We showed how both built-in and custom activities can be designed to run as background tasks or jobs, thus broadening the spectrum of applications.</li></ol><p>Through this exploration, I hope to have shed light on the versatility and power of Elsa 3.0, opening up avenues for innovative workflow solutions across various industries and applications.</p>
