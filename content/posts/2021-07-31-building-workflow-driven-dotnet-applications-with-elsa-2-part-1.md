features:</p><ul><li>Ability to upload a document and specify the type of document.</li><li>The document metadata is stored in a SQLite database while the physical file is stored on disk.</li><li>The application executes any &amp; all workflows associated with the selected document type.</li></ul><p>The uploaded document gets processed by any workflows in the system configured to handle this type of document. A workflow might compress the file and send it as an attachment, request a review from a person, compute a hash and store it on a blockchain, and so on and so forth. The possibilities are endless, and workflows make it easy to quickly setup new processes and change existing ones.</p><p>The following screen recording provides a quick glance of the final result.</p><figure><img data-image-id="1*7pE84agfC_YrqYhKNfgpzg.gif" data-width="2091" data-height="1677" src="https://cdn-images-1.medium.com/max/800/1*7pE84agfC_YrqYhKNfgpzg.gif"></figure><p>For this series we will be implementing the following document types and corresponding workflows:</p><ul><li>Change Request</li><li>Leave Request</li><li>Identity Verification</li></ul><h3>The Workflows</h3><p>The following sections describe the functionality of each workflow. Although the workflows merely serve as an example, the point is to demonstrate key-principles and how one might implement real-world scenarios by creating custom activities that connect to services, databases and APIs, perform long-running jobs, extend JavaScript with custom functions, pass along information through activities and implement long-running processes that involve the user.</p><p>Let’s go over each one of them from a high-level perspective.</p><h4>Change Request Workflow</h4><p>This workflow handles documents of type <em>Change Request</em> and will simply zip the received document file and send it as an attachment, after which the document will be marked as <em>archived</em>.</p><p>The workflow looks like this:</p><figure><img data-image-id="1*oNsAFjIm82gDjoTPxhKDCg.png" data-width="1077" data-height="1782" src="https://cdn-images-1.medium.com/max/800/1*oNsAFjIm82gDjoTPxhKDCg.png"></figure><p>This workflow does not contain any control flow logic — it’s a simple sequence of steps. We’ll discuss each activity separately.</p><h4>Leave Request Workflow</h4><p>This workflow handles documents of type <em>Leave Request</em>. Leave requests can be submitted to the application of you want to e.g. go on a holiday. The workflow will send an email to HR and await an <em>Approve</em> or <em>Reject</em> signal before continuing. At the end, the document will be archived.</p><figure><img data-image-id="1*7YZXcDZCfZUh4-WCxFjW1w.png" data-width="1959" data-height="4061" src="https://cdn-images-1.medium.com/max/800/1*7YZXcDZCfZUh4-WCxFjW1w.png"></figure><h4>Identity Verification Workflow</h4><p>This workflow handles documents of type Identity Verification, and is similar to the Leave Request workflow. The main difference is that this one will include a custom activity that performs a background task using Hangfire, during which the workflow will be suspended. Once the background task completes, the workflow will be loaded back into memory and resume execution.</p><figure><img data-image-id="1*SxmMzyWJaeloL7fS7EvYqg.png" data-width="3136" data-height="6403" src="https://cdn-images-1.medium.com/max/800/1*SxmMzyWJaeloL7fS7EvYqg.png"></figure><h3>The Activities</h3><p>Let’s go over each activity used in the various workflows.</p><h4>Get Document</h4><p>A custom activity that has a single input property called <code>DocumentId</code>.</p><p>The document ID is used to retrieve the document record from the database. This document contains the file name of the uploaded file and is used to load the physical file into memory as a <code>Stream</code>.</p><p>Both the document and the file stream are then made available as the <strong>output</strong> of the activity.</p><p>Subsequent activities can then use this output. For example, the <strong>Send Email</strong> activity can be configured to send the file as an attachment.</p><h4>Archive Document</h4><p>A custom activity that has a single input property called <code>Document</code> of type <code>Document</code>. Notice that this is different when compared to the <strong>Get Document</strong> activity, which only needs a <em>document ID</em>. Therefore, these activities work together: first we load a document, and then we have activities that do something with this document.</p><blockquote>We could have designed this activity to also accept just a document ID and have the activity load the document, but the reason we’re doing it differently here is to demonstrate that we can have activities provide any type of output such as complex objects and process them in a variety of ways.</blockquote><h4>Zip File</h4><p>A custom activity that compresses an input stream into an output stream.</p><h4>Update Blockchain</h4><p>A custom activity that pretends it will store a file signature on some blockchain. The real purpose of this activity is to demonstrate how one might implement an activity that starts a background task using <a href="https://www.hangfire.io/" rel="noopener" target="_blank">Hangfire</a> while the workflow is suspended and then resuming the workflow once the job completed.</p><h4>Send Email</h4><p>An activity provided by the <a href="https://www.nuget.org/packages/Elsa.Activities.Email/" rel="noopener" target="_blank">Elsa.Activities.Email</a> package that can send email messages.</p><h4>Fork</h4><p>An activity provided by <a href="https://www.nuget.org/packages/Elsa.Core/" rel="noopener" target="_blank">Elsa.Core</a> that splits workflow execution in multiple branches. Each of these branches will execute one after another. A fork is typically used when you want to do work in parallel and/or if you want to wait for (one of) multiple triggers simultaneously. A common example is that of a race condition between waiting for user input within a specified amount of time and waiting for a timeout event.</p><h4>Join</h4><p>An activity that joins workflow execution back into a single branch. It is used in combination with the Fork activity and can have one of two modes:</p><ul><li>Wait All</li><li>Wait Any</li></ul><p>The <em>Wait All</em> mode will cause the <em>Join</em> activity to wait for all inbound branches to enter the <em>Join</em> activity. When there are no blocking activities in any of the branches, the <em>Join</em> activity will simply continue execution. But if there are any number of branches blocked, the activity will continue only once each blocking activity is resumed. This mode is useful for scenarios where you for example need to wait for all parties to sign a document before continuing.</p><p>This is in contrast to the <em>Wait Any</em> mode, which will cause the Join activity to continue as soon as any one inbound branch enters. This mode is useful for scenarios where there’s a user action to be taken within a given amount of time, a race condition modeled using for example the <em>Timer</em> activity and the <em>Signal Received</em> activity.</p><p>The <em>Leave Request</em> and <em>Identity Verification</em> both demonstrate the use of the <em>Fork</em> and <em>Join</em> activity using the <em>Wait Any</em> mode.</p><p>Any blocking activities between the <em>Fork</em> and <em>Join</em> activities will be cleared. This means that as soon as the Join activity continues execution, any trigger like timers and signals will no longer be handled.</p><h4>Signal Received</h4><p>An activity acting as a workflow trigger. When encountered, the activity will <em>suspend</em> the workflow until a certain signal is received. When the correct signal is received, the workflow is resumed.</p><p>A common use case is waiting for some user input to be received, as we will see.</p><h4>Write Line</h4><p>A simple activity provided by the <a href="https://www.nuget.org/packages/Elsa.Activities.Console/" rel="noopener" target="_blank">Elsa.Activities.Console</a> package. This activity takes a single string input and writes it to standard out (typically the console).</p><p>This activity is often used for testing out certain pathways in a workflow or as a (temporary) stand-in while the “real” activity is being developed.</p></div></div></section><section><div><div><h3>Next</h3><p>In <a href="https://sipkeschoorstra.medium.com/building-workflow-driven-net-applications-with-elsa-2-part-2-ab0ffae91b95" rel="noopener" target="_blank">the next part</a>, we will scaffold a new .NET solution with basic workflow capabilities. Subsequent parts will tackle implementing custom activities, extensibility, the demo workflows and all other nitty-gritty details we need.</p></div></div></section>
