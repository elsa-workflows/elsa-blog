<section><div><div><h3>Building multi-tenant ASP.NET Core Applications with the Orchard Core Framework</h3><figure><img data-image-id="1*M3xZYaYvm522_X30EFvmaw.jpeg" data-width="1024" data-height="512" src="https://cdn-images-1.medium.com/max/800/1*M3xZYaYvm522_X30EFvmaw.jpeg"></figure><p>As part of a big project we’re working on at <a href="https://www.rtl.nl/over/..en-ontdek/WEggaCcAABI3JRF9" rel="noopener" target="_blank">RTL Netherlands</a>, we decided to implement a microservice-oriented architecture using ASP.NET Core. One of the requirements were that these microservices should be multi-tenant.</p><p>As you’ll come to see, implementing a multi-tenant service could not be simpler with the help of the Orchard Core Framework library.</p><p>And that’s precisely what I’m going to show you in this article: I’ll take you step-by-step to implementing a multi-tenant ASP.NET Core application using Orchard Core Framework.</p><h3>Orchard Core Framework</h3><p><a href="https://blogs.msdn.microsoft.com/webdev/2017/11/21/orchard-core-beta-annoucement/" rel="noopener" target="_blank">Orchard Core</a> is an open source application framework that is built on top of <a href="https://docs.microsoft.com/en-us/aspnet/core" rel="noopener" target="_blank">ASP.NET Core</a>, and is a complete rewrite of Orchard 1.</p><p>Orchard’s original promise was to <a href="https://archive.codeplex.com/?p=orchard" rel="noopener" target="_blank">create shared components for building ASP.NET applications and extensions</a>. Although Orchard version 1 did not quite deliver on that, Orchard Core makes it right, completely and wholly.</p><p>One of Orchard’s best design features has always been its modularity &amp; multi-tenancy. However, in order to take advantage of that, your application would have to <em>be</em> Orchard. But in Orchard Core, it is the other way around: you create your own ASP.NET Core application, and reference only those Orchard packages that you actually need.</p><p>As it turns out, multi-tenancy is at the heart of Orchard Core and made available as a package, allowing us to write modular, multi-tenant applications without having to take any dependencies on the vast array of other functionality it provides.</p><h3>Multi Tenancy</h3><p>A multi tenant application is an application that can serve multiple tenants from a single code base.</p><p>There are compelling advantages to implementing a multi-tenant architecture. Because you can serve multiple tenants from a single code base, it allows for high-density hosting and potentially share the same database. Each tenant has its own configuration, such as connection strings, workflows, extensions, domain name, and so forth.</p><h4>Benefits of a Multi Tenant Architecture</h4><p>Imagine you have an application implemented as a set of microservices for Customer A. Suddenly there is Customer B. Without a multi-tenant architecture, you would need to deploy the entire application and basically have two copies in the cloud. Perhaps this is fine for two customers. But let’s say things are going really well, and before you know it, you have hundreds or even thousands of customers using your cloud service! Maintenance would be a potential nightmare. And then there’s the number of resources that you must consider (costs) for each instance of your application.</p><p>On the other hand, had you implemented the solution using a multi-tenant architecture, not only would deployment be a breeze, think about all the resources you would be saving. In most cases, the opportunity for system reuse between tenants will be close to 100%.</p><p>If you’re not yet convinced why multi-tenancy is a must, I encourage you to read <a href="https://medium.com/u/bb38f563f49f" data-anchor-type="2" data-user-id="bb38f563f49f" data-action-value="bb38f563f49f" data-action="show-user-card" data-action-type="hover" target="_blank">Jakob Rodseth</a>’s article on <a href="https://medium.com/@Integral_io/elegant-multi-tenancy-for-microservices-part-i-why-bother-a88234f9d293" target="_blank">Elegant Multi-Tenancy for Microservices — Part I: Why Bother?</a></p><p>Now that you’re convinced, let’s see what it would take to write a multi-tenant microservice in ASP.NET Core.</p></div></div></section><section><div><div><h3>Walkthrough — Building a Multi Tenant Application</h3><p>In this walkthrough, we’re going to see what it takes to create a multi-tenant application in ASP.NET Core.</p><p>The application will be simple. First, we will implement a middleware function as a lambda that writes a simple string to the response stream. This string will be tenant-specific. Although the application will be simple, it will be easy to imagine how each tenant could have its own configuration, including database connection strings, API keys, and so forth.</p><p>Finally, I will introduce you to a new concept called <strong>features</strong>. A feature is associated to a piece of functionality. Partitioning your service into small features allow you to now only have tenant-specific configuration, but also tenant-specific functionality.</p><p>Let’s have a look!</p><h4>Installing the .NET Core SDK</h4><p>First of all, make sure you have downloaded and installed the latest <a href="https://www.microsoft.com/net/download" rel="noopener" target="_blank">.NET Core SDK</a> (2.1.301 at the time of writing).</p><p>Although I’m using Windows, you can develop ASP.NET Core applications on Mac OS and Linux just as well.</p><p>If you installed .NET Core SDK before, but you’re unsure which version you installed, open a command prompt and type:</p><p><em>dotnet --version</em></p><p>On my machine, it looks like this:</p><figure><img data-image-id="1*Ukh9GhcI7hA-Fkta_Lm_sQ.png" data-width="775" data-height="254" src="https://cdn-images-1.medium.com/max/800/1*Ukh9GhcI7hA-Fkta_Lm_sQ.png"><figcaption><strong>dotnet --version</strong> reveals the installed .NET Core SDK version</figcaption></figure><h4>Creating the Project</h4><p>Create a new folder on your hard drive and execute the following command within that folder:</p><p><em>dotnet new web</em></p><p>That will create a new ASP.NET Core project for us using the <strong>web</strong> template to start with. To see a complete list of available templates, <a href="https://docs.microsoft.com/en-us/dotnet/core/tools/dotnet-new?tabs=netcore21" rel="noopener" target="_blank">click here</a>.</p><p>The folder should now contain the following files and folders:</p><figure><img data-image-id="1*c0PaadcOhVN40WVhG3Dqlw.png" data-width="233" data-height="206" src="https://cdn-images-1.medium.com/max/800/1*c0PaadcOhVN40WVhG3Dqlw.png"><figcaption>The files and folders created by the <strong>web</strong> template</figcaption></figure><h4>Running the Application</h4><p>The <strong>web</strong> template created a <strong>Startup</strong> class that configures a “Hello World” message when we navigate to the web server. You can try it out by typing the following command:</p><p><em>dotnet run</em></p><p>Notice that a web server was launched listening on ports 5000 (HTTP) and 5001 (HTTPS). Surely enough, when you launch a webbrowser and navigate to <a href="https://localhost:5001," rel="noopener" target="_blank">https://localhost:5001,</a> you will see the following output:</p><figure><img data-image-id="1*bq0RlN7wm5MGMCzZnqS9pQ.png" data-width="643" data-height="225" src="https://cdn-images-1.medium.com/max/800/1*bq0RlN7wm5MGMCzZnqS9pQ.png"></figure><p>Now that we’ve seen that the application functions correctly, let’s go ahead and turn it into a multi-tenant application.</p><h4>Adding the OrchardCore.Application.Targets NuGet Package</h4><p>In order to have Orchard turn our application into a multi-tenant application, we first need to reference the OrchardCore.Application.Targets package. To do so, open <strong>MultiTenantApp.csproj</strong> and add the following line:</p><p><code><em>&lt;PackageReference Include=”OrchardCore.Application.Targets” Version=”1.0.0-beta2–67531&quot; /&gt;</em></code></p><h4>Updating Startup.cs</h4><p>Next, we’ll add OrchardCore to the pipeline. Open <em>Startup.cs</em> and add the following line to the (empty) <em>ConfigureServices</em> method:</p><p><code>app.services.AddOrchardCore().WithTenants();</code></p><p>Replace the lines in the <em>Configure</em> method that adds the Hello World message to the request pipeline with the following line:</p><pre>app.UseOrchardCore(a =&gt; a.Run(async context =&gt;<br>{<br>   // ShellSettings provide the tenant&#39;s configuration.           <br>   var shellSettings = <br>      context<br>         .RequestServices<br>         .GetRequiredService&lt;ShellSettings&gt;();                                </pre><pre>   // Read the tenant-specific custom setting.<br>   var customSetting = shellSettings.Configuration[&quot;CustomSetting&quot;];                 </pre><pre>   // Write the value of the custom setting to the response stream.                  <br>   await context.Response.WriteAsync(customSetting);<br>}));</pre><p>That will add the necessary middleware that implements multi-tenancy.</p><p>The complete file right now should look like this:</p><figure><script src="https://gist.github.com/sfmskywalker/b4a3791b519566e099171a1283117d9b.js"></script></figure><p>Notice that the middleware function is defined on the <code>IApplicationBuilder</code> <code>a</code> variable within the <code>UseOrchardCore</code> call. This is important, as it ensures our middleware is multi-tenant. The <code>context</code> argument will be within the context of the requested tenant. This means that whenever we resolve a service from that context, it will contain only services scoped to that tenant.</p><h4>Creating tenants.json</h4><p>Create a new file in the root called <strong>tenants.json</strong> file. You can also provide an actual domain name per tenant. This information is used during the HTTP request to determine to which tenant the HTTP request applies.</p><p>Also notice that I included a custom setting called <code>&quot;CustomSetting&quot;</code>. When you run the project now and navigate to <a href="https://localhost:5001/customer-a." target="_blank">https://localhost:5001/customer-a</a>, you should get the following output:</p><figure><img data-image-id="1*M46bZzw62H5xe1jX135_yQ.png" data-width="412" data-height="149" src="https://cdn-images-1.medium.com/max/800/1*M46bZzw62H5xe1jX135_yQ.png"><figcaption>Accessing the CustomerA tenant</figcaption></figure><p>And as expected, when you navigate to <a href="https://localhost:5001/customer-a." target="_blank">https://localhost:5001/customer-</a>b, you will see:</p><figure><img data-image-id="1*RshaGkLft-9gH3wfgxxmlw.png" data-width="406" data-height="167" src="https://cdn-images-1.medium.com/max/800/1*RshaGkLft-9gH3wfgxxmlw.png"></figure><p>And there you have it — a multi-tenant application!</p></div></div></section><section><div><div><h4>Features</h4><p>The ability to have tenant-specific configuration is all you need in many scenarios, but it doesn’t have to stop there. There may be occasions where you need to enable additional, or perhaps even modified behavior to a given tenant.</p><p>For example, your application could provide a service called <code>DefaultService</code> that implements <code>ISomeService</code> and is used across tenants. But for some tenants, you might need different behavior. In other words, you may want to provide a different implementation of <code>ISomeService</code> for those tenants.</p><p>That is where Features come into play.</p><p>In order to associate functionality with a tenant, we can configure a tenant with a set of so-called “features”.</p><p>A feature is declared using an assembly-level attribute called <code>Feature</code>.
