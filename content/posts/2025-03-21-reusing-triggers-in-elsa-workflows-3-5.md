<h4>Elsa 3.5 Preview</h4><figure><img alt="" src="https://cdn-images-1.medium.com/max/1024/1*e3HFPUDKgWuLPxR7j9-gWw.png" /><figcaption>Event driven workflows made easy with Elsa Workflows</figcaption></figure><p>Elsa Workflows is getting a new feature in version <strong>3.5</strong> that makes it easier to build custom trigger-based activities. With new reusable base classes, you can implement activities like <em>timers</em>, <em>event listeners</em>, and <em>HTTP endpoints</em> without dealing with low-level infrastructure.</p><p>This post walks through what’s new and how you can start using these features today — even though Elsa 3.5 is still in preview.</p><h3>Quick Note on Versioning</h3><p>These features are part of the <strong>Elsa 3.5.0 preview</strong> release. If you’re using Elsa 3.3 or 3.4, you won’t see them yet.</p><p>To try it out now:</p><ul><li>Reference the 3.5.0-preview.xxx packages.</li><li>Use the <strong>preview feed from </strong><a href="https://docs.elsaworkflows.io/getting-started/packages#previews"><strong>feez.io</strong></a>.</li></ul><p>Check the <a href="https://docs.elsaworkflows.io/getting-started/packages#previews">official docs</a> for instructions on how to configure your NuGet feed to include preview versions.</p><h3>What’s New?</h3><p>Elsa 3.5 introduces base classes that simplify common trigger patterns:</p><ul><li>EventBase&lt;T&gt; – for event-driven activities</li><li>TimerBase – for recurring time-based logic</li><li>HttpEndpointBase – for triggering workflows via HTTP</li><li>DelayFor(...) – for pausing execution inside a custom activity</li></ul><p>These help you avoid repetitive infrastructure code, keeping your activities focused and easier to maintain.</p><h3>Event-Based Activities with EventBase&lt;T&gt;</h3><p>This base class is for workflows that need to react to named events — like something happening in your application or an external system.</p><pre>public class CustomEvent : EventBase&lt;object&gt;<br>{<br>    protected override string GetEventName(ExpressionExecutionContext context) =&gt; &quot;MyEvent&quot;;<br><br>    protected override void OnEventReceived(ActivityExecutionContext context, object? eventData)<br>    {<br>        Console.WriteLine(&quot;Event received with data: &quot; + eventData);<br>    }<br>}</pre><p>This pattern is useful for workflows that need to respond to domain events, integrations, or other asynchronous triggers.</p><h3>Recurring Activities with TimerBase</h3><p>For periodic logic like polling or heartbeats, use TimerBase.</p><pre>public class CustomTimer : TimerBase<br>{<br>    protected override TimeSpan GetInterval(ExpressionExecutionContext context) =&gt; TimeSpan.FromSeconds(5);<br><br>    protected override void OnTimerElapsed(ActivityExecutionContext context)<br>    {<br>        Console.WriteLine(&quot;Timer elapsed&quot;);<br>    }<br>}</pre><p>It’s a quick way to schedule regular workflow execution with a custom activity other than the built-in Timer activity.</p><h3>HTTP-Triggered Workflows with HttpEndpointBase</h3><p>You can define HTTP endpoints directly in your activities — no extra routing or middleware needed.</p><pre>public class CustomHttpEndpoint : HttpEndpointBase<br>{<br>    protected override HttpEndpointOptions GetOptions() =&gt; new()<br>    {<br>        Path = &quot;my-path&quot;,<br>        Methods = [HttpMethods.Get]<br>    };<br><br>    protected override async ValueTask OnHttpRequestReceivedAsync(ActivityExecutionContext context, HttpContext httpContext)<br>    {<br>        httpContext.Response.StatusCode = 200;<br>        await httpContext.Response.WriteAsync(&quot;Hello World&quot;, context.CancellationToken);<br>    }<br>}</pre><p>This is a useful way to create lightweight API endpoints or webhook handlers that connect directly into your workflows.</p><h3>Delays with DelayFor</h3><p>For workflows that need to pause and resume later, you can use DelayFor inside any activity.</p><pre>public class CustomDelay : Activity<br>{<br>    protected override ValueTask ExecuteAsync(ActivityExecutionContext context)<br>    {<br>        context.DelayFor(TimeSpan.FromSeconds(5), OnDelayElapsedAsync);<br>        return default;<br>    }<br><br>    private async ValueTask OnDelayElapsedAsync(ActivityExecutionContext context)<br>    {<br>        Console.WriteLine(&quot;Delay elapsed&quot;);<br>        await context.CompleteActivityAsync();<br>    }<br>}</pre><p>This helps keep delay logic contained within the activity that needs it.</p><h3>Summary</h3><p>The new trigger base classes in <strong>Elsa 3.5</strong> let you:</p><ul><li>Build event, timer, and HTTP activities without wiring up the internals.</li><li>Keep custom logic clean and focused.</li><li>Compose more maintainable workflows.</li></ul><blockquote><strong><em>Want to try it now?</em></strong><em><br> Grab the </em><em>3.5.0-preview.xxx packages from </em><a href="https://docs.elsaworkflows.io/getting-started/packages#previews"><em>feedz.io</em></a><em> and start experimenting with reusable triggers today.</em></blockquote><p>If you’re building workflows that rely on timers, HTTP calls, or events, these new base classes can save you time and help keep things organized.</p>
